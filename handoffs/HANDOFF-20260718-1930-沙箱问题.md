### 1.2 根本原因

- 对 `D:\taiyang`（项目目录之外）的读写被重定向到沙箱隔离区
- git push 通过网络绕过沙箱，推到了真实 GitHub
- 但本地工作目录文件没同步到用户物理 D 盘

### 1.3 解决方案

**步骤 1：在物理 D 盘同步 GitHub**
```powershell
cd D:\taiyang
git pull origin master
```
在 Windows 终端或 TRAE IDE 终端执行（不能在 TRAE Work 沙箱里执行）。这会把所有最新代码拉到物理 D 盘，包括知识库 30、31、AGENTS v2.6、Phase 4.1 代码修复等。

**步骤 2：切换开发环境**
- **推荐**：下载 [TRAE IDE](https://www.trae.cn/ide/download)，打开 `D:\taiyang` 作为项目目录。IDE 沙箱允许项目目录内读写，文件直接落物理盘
- **备选**：TRAE Work 切换到 Code 模式（Windows 上 Code 模式沙箱策略不同，项目目录可写）
- **不推荐**：继续用 Work 模式——沙箱隔离无法关闭，每次文件修改都无法落物理盘

**步骤 3：验证文件同步**
拉取后在 `D:\taiyang\知识库\` 确认能看到 `30-碰撞层与碰撞遮罩.md` 和 `31-速度推力与软推力.md`。在 Obsidian 里按 F5 或重启确认。

## 四、重要的执行教训（给下一位 AI 助手）

### 4.1 沙箱认知盲点

TRAE Work 桌面版 Windows + Work 模式的沙箱是**独立虚拟化环境，无法关闭**。对项目目录之外的路径（如用户的 `D:\taiyang`）读写会被重定向到沙箱隔离区。**PowerShell 能读到文件 ≠ 文件真的写到了用户物理盘**。必须用双向验证（用户建文件我读 + 我建文件用户看）确认沙箱边界。

### 4.2 图片识图能力

当前模型（GLM-5.2）**没有图片识图能力**。如果用户发图片，必须直接告知"我看不了图片，请用文字描述"，不能假装看懂编造分析。

### 4.3 执行滑坡问题

用户对"执行滑坡"零容忍。之前的滑坡包括：
- 知识库笔记不符合 7 段标准（缺代码执行流程/优势/关联段）
- 用专业术语没立即用大白话解释（如"软推力"）
- 闸门 A/B/C 有机制但跳过执行
- 技术选型无论证（方案 A 直接选没做架构地基三问）

**解决方案**：项目根目录有 `每次对话提示词.md`（146 字），用户每次开新对话会贴给 AI。三级铁律：句子级（术语自检）/ 任务级（闸门A前B后）/ 选型级（架构地基三问）。

### 4.4 AGENTS.md v2.6 核心规矩

- **§0.5 架构地基优先原则**（凌驾所有规则）：技术选型必答三问——MVP够用/加系统不重构/升级地基不改
- **§1 代码地基三规矩**：逻辑显示分离 / 数据驱动 / 统一存档
- **§1.4 知识库 7 段标准**：概念(含比喻)→功能→运作(含代码逐行)→原理→优势→场景→关联段
- **§1.5 三道闸门**：A 任务前打印清单 / B 任务后 7 问 / C 差异审计 5 问
- **§1.6 三项一致**：docs/数据/代码同步改不延后
