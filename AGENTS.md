## 1. 必须遵守的代码规矩（地基三规矩）

以下三条从第一行代码起强制遵守，违反即为错误：

### 1.1 游戏逻辑与画面显示分离

- 游戏规则（伤害计算、等级检查、物品能否使用、宠物捕捉概率）放在独立脚本
- **永远不直接操作画面**（不直接改血条、不直接播动画）
- 将来搬上服务器不用改代码

**示例**：
```
✅ 正确：
  attack_manager.gd → 计算伤害 → 返回伤害值
  enemy.gd → 收到伤害值 → 扣血 → 通知动画播放

❌ 错误：
  attack_manager.gd → 直接改 enemy 的血条 + 直接播动画
```

### 1.2 数据表驱动，不写死代码

- 怪物属性、物品数据、食谱、技能参数等全部走 JSON 文件
- 加新内容 = JSON 加一行，不改代码
- **禁止**在 .gd 文件中写 `if name == "红尾蝎": damage = 15` 这类硬编码
- 所有数据文件必须落在 §1.4 列出的 JSON 清单之一里，不可新建未登记的 JSON

### 1.3 存档用统一接口

- 所有存档操作走 `SaveManager` 统一接口
- `SaveManager.save()` / `SaveManager.load()` / `SaveManager.delete()`
- 底层实现（JSON / SQLite / 未来 PostgreSQL）可换，上层调用不变

### 1.4 数据文件清单（地基规矩 2 的落地）

所有"可变的数字"必须落在以下 JSON 之一：

| 文件 | 装什么 | MVP 状态 |
|------|--------|:---:|
| `data/monsters.json` | 怪物属性/掉落/AI 模板 | ✅ |
| `data/items.json` | 物品名字/描述/稀有度（含重要物品太阳印记、宠物蛋） | ✅ |
| `data/equipment.json` | 装备攻防/耐久/附带技能 | ✅ |
| `data/skills.json` | 技能伤害/CD/属性 | ✅ |
| `data/levels.json` | 等级 XP 表（BASE=80, GROWTH_RATE=1.5） | ✅ |
| `data/recipes.json` | 食谱/配方（材料+产出） | ✅ |
| `data/npcs.json` | NPC 名字/位置/对话 ID | ✅ |
| `data/quests.json` | 任务触发条件/奖励 | ✅ |
| `data/timers.json` | 等待时长（种植/炼金，留接口） | ✅ 占位 |
| `data/maps.json` | 地图出口/安全点/怪物刷新点 | ✅ |
| `data/pets.json` | 宠物属性/喂食材料/突破加成 | ✅ |
| `data/characters.json` | 角色初始属性（HP/MP/属性点比例） | ✅ |
| `data/dialogues.json` | NPC 对话文本（多语言扩展接口） | ✅ 占位 |
| `data/mails.json` | 邮件配置（系统邮件/物品附件/过期） | ✅ |
| `data/sounds.json` | 音效/音乐配置（BGM 路径/音效路径/loop） | ✅ |
| `data/palettes.json` | 美术调色板（16~32 色） | ⏳ 美术阶段 |
| `data/save_slots/` | 存档（动态生成，不进版本管理） | 动态 |

**新增 JSON 文件必须先在 §1.4 登记，否则视为违规**。

### 知识库运维协议

每接触一个新领域/概念/工具，AI 必须**自发按统一标准**在 `知识库/` 下新建或补充笔记：

**统一标准**：概念 → 功能 → 运作方式（含具体例子）→ 原理 → 优势 → 使用场景（含对接） → "关联"段（双链到相关笔记）。

- 零基础用户读得懂为标准
- 每个概念必须有生活类比
- 每个机制必须有"反例 vs 正例"对照
- 每篇笔记末尾必须有"关联"段指向相关概念（形成知识网）

**触发条件（必须立即写入知识库的 9 种时机）**：
1. 用户说"这是什么"或问任何新名词 → 立即写
2. AI 用了一个新术语用户没听懂 → 立即写
3. 引入新技术/新工具 → 按 §15 审计同时写
4. 调试代码时遇到不懂的现象 → 解释清楚后写
5. 用户操作 Obsidian/Godot 等工具问了"怎么做" → 同步写
6. 完成一个阶段做交接 → 总结该阶段新学到的一并写
7. 你和 AI 出现认知摩擦（你以为 A 实际是 B）→ 澄清后立即写
8. AI 发现自己在重复解释同一概念（第二次讲）→ 第二次必写
9. **⚠️ 代码报错 / 修 Bug / 解释代码片段 → 强制触发**：
   - AI 必须用小白能懂的生活比喻解释报错根因
   - 解释每段代码每行的具体作用、格式、词的含义、逻辑
   - 修复后立即按统一 7 段标准补知识库（概念/功能/运作方式/原理/优势/场景/关联）
   - **严格执行不停留"快速修复"模式**——修代码与补知识库不可分割

**知识库不是 AI 必读**，是用户的查询资源。但保持它跟项目同步是 AI 的责任。

**强制流程：报错三步流程**（不可跳过）：

每次用户贴报错，AI 必须按顺序执行：
1. **解释根因**：用生活比喻讲清楚报错为什么发生（用词要小白能懂）
2. **修复**：明确指出哪行哪词错了，改成正确版本
3. **补知识库**：按 7 段统一标准新建或补充知识库笔记，并更新知识库索引

**禁止行为**：
- ❌ 只修代码不解释（视为绕过流程）
- ❌ 只解释不补知识库（视为绕过流程）
- ❌ 用专业术语解释给小白（视为无效解释）

**知识库 5 弊端与 5 解法**（防塌方）：

| # | 弊端 | 解法 |
|---|------|------|
| 1 | 碎片化无结构 | 索引页（00-索引.md）强制维护，每加一篇笔记同步更新 |
| 2 | 同概念多处定义 | 一概念一家"家"——别篇用 `[[文件名]]` 跳转，不复制 |
| 3 | AI 写得太学术 | 必须含生活类比 + 反例 vs 正例对照，否则退回重写 |
| 4 | 知识库和 docs 重复 | docs=AI 干活必读的设计真值；知识库=用户学概念的辅助。一概念可能在两处有，但 docs 写"设计规则"，知识库写"这是什么+为什么" |
| 5 | 链接断链 | AI 写新笔记时试一遍双链文件名是否存在，不存在用"[待写]"占位 |

## 19. 任务拆分度判定 3 问（防过细 / 防过粗）

每次拆任务前，AI 必答 3 问：

| 问 | 通过条件 |
|---|---------|
| ① 当前任务用户有具体执行依据吗？（手感有反馈、错误有具体来源）| 有 → 通过；无 → 不拆细 |
| ② 拆出来的每一项，用户做完后能独立验证吗？| 能 → 通过；不能 → 不拆细 |
| ③ 拆 1 项 ≤2 小时 + 可以独立运行测试？| 是 → 通过；否则不拆细 |

三个都 ✅ → 拆细
任一 ❌ → 等再做一步再拆

**总原则**：沉浸式分阶段细化。当前阶段做到能验证就拆细；下个阶段开始前再拆细。不一次性全拆细——避免未来阶段开工时发现预先拆的颗粒度不适用。

任何 AI 会话开始时**第一个动作**必须是：

1. 读 `AGENTS.md`（本文件）

**禁止跳过此协议直接动手**。新 AI 会话、换模型、隔几天回来都必须执行。

## 1.6 三项一致原则（防塌方方案 2）

任何提交必须同时满足：

| 项 | 是什么 |
|----|--------|
| ① 设计文档（docs/） | "这个功能该是什么样" |
| ② 数据文件（data/*.json） | "这个功能的数值" |
| ③ 代码（scripts/*.gd） | "这个功能的实现" |

改了一项必须同步另外两项（如果涉及）。

**提交信息格式**：`[变更类型] 改了什么 + 文档是否同步`

```
✅ [新增] 钓鱼系统，已更新 05-副职业.md 和 data/fishing.json
❌ [新增] 钓鱼系统  ← 没说文档是否同步
```

未同步文档的提交不算完成。

## 1.7 新增系统协议（防塌方方案 5）

引入任何**新系统/新技术/新概念**前，必须先回答 4 个问题：

1. 这件事在 `docs/` 或下方"能力总目录"里提过吗？
2. 它会影响哪些现有系统？（查能力总目录的依赖/被依赖列）
3. 它需要新增哪个 JSON 文件？（查 §1.4 数据文件清单）
4. 三项一致吗？（文档/数据/代码同步）

答完 4 问，向用户复述，得到确认后才能动手。

### 2.2 MVP 只做

- 5 张地图（玩家家 + 新手村 + 野外 A + 野外 B + 小城镇）
- 家（院子+屋内）：床/存档点 + 储物箱 + 烹饪灶台，未来职业设施预留位置
- 剑 1 把（剑士职业）
- 3~4 种怪物 + 1 个 Boss
- 宠物蛋 ×3 种（拿到自动孵化）
- 副职业：钓鱼 + 烹饪（顺序解锁，均需体验任务后正式学）
- 等级 1~15，Lv15 锁经验需转职
- NPC 铁匠（修装备，修复价较高，为未来锻造师副职业铺垫）
- NPC 商店（卖白/绿装 + 基础材料 + 鱼竿鱼饵）
- 主线第一章（约 100~110 分钟，12 个剧情节点）
- 本地存档（自动 + 手动多槽位）
- 太阳印记（剧情视觉，无机制效果）
- 全屏自适应支持
- GameTime 单例（太阳历显示 + 现实等待接口预留）
- IntroSequence（过场控制，文字版先上留 B 方案接口）
- 邮件系统 MVP 版（满包奖励自动入邮件 + 物品附件领取 + 24h 过期）

## 17. AGENTS.md 片段拼装协议（v2.2 新增）

**不允许在用户不知情下擅自拆 AGENTS.md**。这是项目宪法，用户必须确认。

### 拆分方式（参考 coding-agents-setup 的片段拼装思路）

如果用户同意拆，按"阶段拆分"：

```
D:\taiyang\
├── AGENTS-shared.md       ← §0 项目身份 / §1 地基三规矩 / §1.4 数据清单 / §1.5 对话开始协议 / §3 文档管理 / §6 安全 / §16 修改制度（跨阶段共用）
├── AGENTS-mvp.md          ← §2 MVP 范围红线 / §2.3 MVP 验收清单 / P0 漏洞（MVP 期间适用）
└── AGENTS-online.md       ← 联机版新增的内容（MVP 后启动）
```

阶段拆分的好处：
| 好处 | 说明 |
|------|------|
| 每个 AI 只读当前阶段相关部分 | 减少上下文压力 |
| MVP AI 不会被联机内容干扰 | 范围更清晰 |
| 切换阶段只改入口指向 | 平滑过渡 |

入口定义：MVP 阶段 AGENTS.md 顶部显式说明：
```
本阶段规矩来源：
  AGENTS-shared.md   (跨阶段共用)
  AGENTS-mvp.md      (MVP 适用)
将来转联机版时：替换 AGENTS-mvp.md 为 AGENTS-online.md
```

### 是否适用的判断 5 问

用户问"该不该拆"时，AI 必须答 5 问：

1. AGENTS.md 已经多少行？（用 wc -l 实测）
2. 当前阶段是哪个？（MVP / 缩小全内容 / 联机）
3. 跨阶段共用的内容多还是阶段专用内容多？（共用多则可能不必拆）
4. AI 读 AGENTS.md 的上下文消耗是否影响响应质量？（读不下的影响如果小则不必拆）
5. 用户是否有切换阶段的需求？（如果不切阶段保持不拆也行）

答完后给推荐："建议拆" / "不必拆" / "用户自定义"。
