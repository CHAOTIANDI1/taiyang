## 1. 必须遵守的代码规矩（地基三规矩）

以下三条从第一行代码起强制遵守，违反即为错误：

### 1.1 游戏逻辑与画面显示分离

- 游戏规则（伤害计算、等级检查、物品能否使用、宠物捕捉概率）放在独立脚本
- **永远不直接操作画面**（不直接改血条、不直接播动画）
- 将来搬上服务器不用改代码

**示例**：
```
✅ 正确：
  attack_manager.gd → 计算伤害 → 返回伤害值
  enemy.gd → 收到伤害值 → 扣血 → 通知动画播放

❌ 错误：
  attack_manager.gd → 直接改 enemy 的血条 + 直接播动画
```

### 1.2 数据表驱动，不写死代码

- 怪物属性、物品数据、食谱、技能参数等全部走 JSON 文件
- 加新内容 = JSON 加一行，不改代码
- **禁止**在 .gd 文件中写 `if name == "红尾蝎": damage = 15` 这类硬编码

### 1.3 存档用统一接口

- 所有存档操作走 `SaveManager` 统一接口
- `SaveManager.save()` / `SaveManager.load()` / `SaveManager.delete()`
- 底层实现（JSON / SQLite / 未来 PostgreSQL）可换，上层调用不变

## 2. MVP 范围红线

以下内容在 MVP 阶段**禁止实现**，违反即超出范围：

### 2.1 不碰的系统

| 禁止项 | 原因 |
|--------|------|
| 后端/服务器代码 | 联机版才做 |
| 数据库（PostgreSQL/Redis） | 联机版才做 |
| 交易市场频道系统 | 联机版才做 |
| 世界聊天 / PvP / 组队 | 联机版才做 |
| 炼金/锻造/种植/药剂副职业 | MVP 只做钓鱼+烹饪 |
| 法师/弓手职业 | MVP 只做剑士 |
| 副本/秘境 | 主线结束后内容 |
| 季节/天气 | 预留字段，不实现 |
| 宠物进化 | 只有等级系统 |
| 角色立绘 | 纯对话框 |
| 第三方插件（未授权） | 需用户明确同意 |

### 2.2 MVP 只做

- 1 新手村 + 1 野外地图片区 + 1 主城（含 NPC 商店）
- 剑 1 把（剑士职业）
- 3~4 种怪物 + 1 个 Boss
- 宠物蛋 ×3 种（孵化后战斗）
- 副职业：钓鱼 + 烹饪
- 等级 1~10
- 主线第一章（3~5 个任务）
- 本地存档

```

│   ├── enemy/

│   ├── combat/                  # 战斗系统
│   ├── pet/                     # 宠物系统
│   ├── ui/                      # UI 脚本

├── data/                        # 数据表（JSON）
│   ├── monsters.json
│   ├── items.json
│   └── recipes.json

│   ├── sprites/
│   ├── tilesets/
│   └── audio/
└── .gitignore
```

### 4.2 命名约定

| 类型 | 命名风格 | 示例 |
|------|---------|------|
| 脚本文件 | snake_case | `save_manager.gd` |
| 场景文件 | snake_case | `player_scene.tscn` |
| 数据文件 | snake_case + 复数 | `monsters.json` |
| 节点名 | PascalCase | `PlayerCharacter` |
| 函数名 | snake_case | `take_damage()` |
| 常量 | UPPER_SNAKE_CASE | `MAX_HP` |
| 信号 | snake_case | `enemy_died` |

### 4.3 注释规则

- **不加注释除非用户明确要求**
- 代码自解释：函数名就是文档
- 如需解释设计意图，写在 `docs/` 对应文档中，不写在代码里

## 5. 用户交互规矩

### 5.1 用户是零基础

- 解释每个专业术语
- 不假设用户懂得引擎概念（节点、场景、信号等需要解释）
- 每次给代码时解释关键逻辑
- 用户问"为什么"时，用生活类比解释

### 5.2 操作分工

| AI 做 | 用户做 |
|--------|--------|
| 写所有 GDScript 代码 | 粘贴到 Godot 脚本编辑器 |
| 指导节点创建步骤 | 在 Godot 里创建节点 |
| 解释每行代码 | 问不懂的地方 |
| 定位 Bug + 修复 | 贴报错信息 |
| 设计数据表 JSON | —— |
| 指导项目结构 | 在文件管理器创建目录 |

### 5.3 验证规矩

- 每完成一个功能，告诉用户如何验证（运行、操作步骤、预期结果）
- 不在没有验证证据的情况下声称"完成"
- 用户反馈"不对"时，先看代码和数据，再问操作步骤

## 6. 安全规矩

### 6.1 禁止事项

- 不引入未经用户授权的第三方插件/库
- 不修改 `.godot/` 目录内容（引擎自动管理）
- 不在代码中写死 API Key、密码、密钥等敏感信息
- 不删除 Git 历史
- 不强制推送（force push）
- 不修改全局 Godot 配置

### 6.2 Git 规矩

- 每个功能完成后提交一次
- 提交信息用中文，说明改了什么
- 不提交 `.godot/` 目录（已在 .gitignore 排除）
- 不提交个人配置/密钥
