## 1. 必须遵守的代码规矩（地基三规矩）

以下三条从第一行代码起强制遵守，违反即为错误：

### 1.1 游戏逻辑与画面显示分离

- 游戏规则（伤害计算、等级检查、物品能否使用、宠物捕捉概率）放在独立脚本
- **永远不直接操作画面**（不直接改血条、不直接播动画）
- 将来搬上服务器不用改代码

**示例**：
```
✅ 正确：
  attack_manager.gd → 计算伤害 → 返回伤害值
  enemy.gd → 收到伤害值 → 扣血 → 通知动画播放

❌ 错误：
  attack_manager.gd → 直接改 enemy 的血条 + 直接播动画
```

### 1.2 数据表驱动，不写死代码

- 怪物属性、物品数据、食谱、技能参数等全部走 JSON 文件
- 加新内容 = JSON 加一行，不改代码
- **禁止**在 .gd 文件中写 `if name == "红尾蝎": damage = 15` 这类硬编码
- 所有数据文件必须落在 §1.4 列出的 JSON 清单之一里，不可新建未登记的 JSON

### 1.3 存档用统一接口

- 所有存档操作走 `SaveManager` 统一接口
- `SaveManager.save()` / `SaveManager.load()` / `SaveManager.delete()`
- 底层实现（JSON / SQLite / 未来 PostgreSQL）可换，上层调用不变

### 1.4 数据文件清单（地基规矩 2 的落地）

所有"可变的数字"必须落在以下 JSON 之一：

| 文件 | 装什么 | MVP 状态 |
|------|--------|:---:|
| `data/monsters.json` | 怪物属性/掉落/AI 模板 | ✅ |
| `data/items.json` | 物品名字/描述/稀有度（含重要物品太阳印记、宠物蛋） | ✅ |
| `data/equipment.json` | 装备攻防/耐久/附带技能 | ✅ |
| `data/skills.json` | 技能伤害/CD/属性 | ✅ |
| `data/levels.json` | 等级 XP 表（BASE=80, GROWTH_RATE=1.5） | ✅ |
| `data/recipes.json` | 食谱/配方（材料+产出） | ✅ |
| `data/npcs.json` | NPC 名字/位置/对话 ID | ✅ |
| `data/quests.json` | 任务触发条件/奖励 | ✅ |
| `data/timers.json` | 等待时长（种植/炼金，留接口） | ✅ 占位 |
| `data/maps.json` | 地图出口/安全点/怪物刷新点 | ✅ |
| `data/pets.json` | 宠物属性/喂食材料/突破加成 | ✅ |
| `data/characters.json` | 角色初始属性（HP/MP/属性点比例） | ✅ |
| `data/dialogues.json` | NPC 对话文本（多语言扩展接口） | ✅ 占位 |
| `data/mails.json` | 邮件配置（系统邮件/物品附件/过期） | ✅ |
| `data/palettes.json` | 美术调色板（16~32 色） | ⏳ 美术阶段 |
| `data/save_slots/` | 存档（动态生成，不进版本管理） | 动态 |

**新增 JSON 文件必须先在 §1.4 登记，否则视为违规**。

## 1.5 对话开始协议（防塌方方案 1）

任何 AI 会话开始时**第一个动作**必须是：

1. 读 `AGENTS.md`（本文件）
2. 读 `docs/00-总览.md`
3. 读 `docs/12-技术栈与架构.md`（地基规矩细节）
4. 用 5 句话向用户复述当前项目状态
5. 等待用户下一步指示

**禁止跳过此协议直接动手**。新 AI 会话、换模型、隔几天回来都必须执行。

## 1.6 三项一致原则（防塌方方案 2）

任何提交必须同时满足：

| 项 | 是什么 |
|----|--------|
| ① 设计文档（docs/） | "这个功能该是什么样" |
| ② 数据文件（data/*.json） | "这个功能的数值" |
| ③ 代码（scripts/*.gd） | "这个功能的实现" |

改了一项必须同步另外两项（如果涉及）。

**提交信息格式**：`[变更类型] 改了什么 + 文档是否同步`

```
✅ [新增] 钓鱼系统，已更新 05-副职业.md 和 data/fishing.json
❌ [新增] 钓鱼系统  ← 没说文档是否同步
```

未同步文档的提交不算完成。

## 1.7 新增系统协议（防塌方方案 5）

引入任何**新系统/新技术/新概念**前，必须先回答 4 个问题：

1. 这件事在 `docs/` 或下方"能力总目录"里提过吗？
2. 它会影响哪些现有系统？（查能力总目录的依赖/被依赖列）
3. 它需要新增哪个 JSON 文件？（查 §1.4 数据文件清单）
4. 三项一致吗？（文档/数据/代码同步）

答完 4 问，向用户复述，得到确认后才能动手。

## 2. MVP 范围红线

### 2.1 不碰的系统

| 禁止项 | 原因 |
|--------|------|
| 后端/服务器代码 | 联机版才做 |
| 数据库（PostgreSQL/Redis） | 联机版才做 |
| 交易市场频道系统 | 联机版才做 |
| 玩家间邮件 | 联机版才做 |
| 世界聊天 / PvP / 组队 | 联机版才做 |
| 成就奖章系统（发奖时间戳） | 联机版才做 |
| 锻造师副职业 | 用 NPC 铁匠替代 |
| 炼金/种植/药剂副职业 | MVP 只做钓鱼+烹饪 |
| 技能书系统 | V1 技能来源只走武器+护甲+职业天赋 |
| 法师/弓手职业 | MVP 只做剑士 |
| 副本/秘境 | 主线结束后内容 |
| 季节/天气 | 预留字段，不实现 |
| 宠物进化 | 只有等级系统 |
| 角色立绘 | 纯对话框 |
| 太阳印记机制效果 | 仅剧情视觉 |
| 第三方插件（未授权） | 需用户明确同意 |

### 2.2 MVP 只做

- 5 张地图（玩家家 + 新手村 + 野外 A + 野外 B + 小城镇）
- 家（院子+屋内）：床/存档点 + 储物箱 + 烹饪灶台，未来职业设施预留位置
- 剑 1 把（剑士职业）
- 3~4 种怪物 + 1 个 Boss
- 宠物蛋 ×3 种（拿到自动孵化）
- 副职业：钓鱼 + 烹饪（顺序解锁，均需体验任务后正式学）
- 等级 1~15，Lv15 锁经验需转职
- NPC 铁匠（修装备，修复价较高，为未来锻造师副职业铺垫）
- NPC 商店（卖白/绿装 + 基础材料 + 鱼竿鱼饵）
- 主线第一章（约 100~110 分钟，12 个剧情节点）
- 本地存档（自动 + 手动多槽位）
- 太阳印记（剧情视觉，无机制效果）
- 全屏自适应支持
- GameTime 单例（太阳历显示 + 现实等待接口预留）
- IntroSequence（过场控制，文字版先上留 B 方案接口）
- 邮件系统 MVP 版（满包奖励自动入邮件 + 物品附件领取 + 24h 过期）

## 5. 用户交互规矩

### 5.1 用户是零基础

- 解释每个专业术语
- 不假设用户懂得引擎概念（节点、场景、信号等需要解释）
- 每次给代码时解释关键逻辑
- 用户问"为什么"时，用生活类比解释

### 5.2 操作分工

| AI 做 | 用户做 |
|--------|--------|
| 写所有 GDScript 代码 | 粘贴到 Godot 脚本编辑器 |
| 指导节点创建步骤 | 在 Godot 里创建节点 |
| 解释每行代码 | 问不懂的地方 |
| 定位 Bug + 修复 | 贴报错信息 |
| 设计数据表 JSON | —— |
| 指导项目结构 | 在文件管理器创建目录 |

### 5.3 验证规矩

- 每完成一个功能，告诉用户如何验证（运行、操作步骤、预期结果）
- 不在没有验证证据的情况下声称"完成"
- 用户反馈"不对"时，先看代码和数据，再问操作步骤

## 6. 安全规矩

### 6.1 禁止事项

- 不引入未经用户授权的第三方插件/库
- 不修改 `.godot/` 目录内容（引擎自动管理）
- 不在代码中写死 API Key、密码、密钥等敏感信息
- 不删除 Git 历史
- 不强制推送（force push）
- 不修改全局 Godot 配置
- 不在能力总目录外擅自加新系统（见 §1.7）
- 不在数据文件清单外擅自加新 JSON（见 §1.4）

### 6.2 Git 规矩

- 每个功能完成后提交一次
- 提交信息用中文，遵循 §1.6 三项一致原则
- 不提交 `.godot/` 目录（已在 .gitignore 排除）
- 不提交个人配置/密钥
- 不提交 `data/save_slots/`（动态存档）

### 6.3 合规巡检（防塌方方案 7）

每完成一个里程碑（或换会话/换模型）时，AI 必须主动运行一次合规巡检：

```
1. 抽查最近 3 个 commit，文档是否同步更新
2. 抽查 data/ 有没有新建未登记的 JSON
3. 抽查 scripts/ 有没有硬编码 if name==...
4. 抽查 §2.1 MVP 范围有没有越界
5. 抽查 §1.8 能力总目录有没有未登记的新系统
```

发现违规 → 立即向用户报告，等待修复指示，不擅自修复。
