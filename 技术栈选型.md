## 二、为什么是 Godot，不是别的

### 对比表

| 维度 | Godot 4 | Unity | RPG Maker | GameMaker |
|------|---------|-------|-----------|-----------|
| 价格 | 完全免费 | 个人免费，有限制 | 付费 | 付费/免费版受限 |
| 安装包大小 | ~90MB | 数 GB | 几百 MB | ~200MB |
| 学习曲线 | ★★☆☆☆ | ★★★★☆ | ★☆☆☆☆ | ★★★☆☆ |
| 2D 支持 | ⭐⭐⭐⭐⭐ | ⭐⭐⭐ | ⭐⭐⭐ | ⭐⭐⭐⭐ |
| 实时战斗 | ✅ 方便 | ✅ 方便 | ❌ 很难 | ✅ 方便 |
| 自定义 UI | ✅ | ✅ | ❌ | ⭐⭐⭐ |
| 联机功能 | 内置节点 | 需第三方插件 | ❌ | 需插件 |
| AI 代码能力 | ⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ | ⭐⭐ | ⭐⭐⭐ |
| 单人开发友好 | ⭐⭐⭐⭐⭐ | ⭐⭐⭐ | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐ |

### 逐个排除

- **RPG Maker**：做回合制 RPG 很爽，但你要做实时动作战斗——RPG Maker 的实时战斗需要大量"黑科技"插件，底层不支持，直接排除。
- **GameMaker**：能做，但生态较小，AI 对它的代码训练数据少，AI 辅助效果打折扣。
- **Unity**：功能最全、教程最多、AI 代码能力最强。但你是一个人 + 零基础 + 做 2D 像素 RPG。Unity 为了这个目标开机就要加载几 GB 的工具链，90% 的功能你用不到。C# 入门门槛比 GDScript 高一截。**刀太大，不适合切水果。**
- **Godot 4**：轻量（90MB）、免费无限制、2D 是原生强项、语言简单、内置联机节点。一个人打开就能干活。而且 AI 对 Godot/GDScript 的掌握已经足够辅助你开发。

### GDScript 长什么样（给你一个直观感受）

```gdscript
# 这是 Godot 的脚本语言，对零基础非常友好

extends CharacterBody2D

var speed = 200
var health = 100

func _physics_process(delta):
    var direction = Input.get_vector("ui_left", "ui_right", "ui_up", "ui_down")
    velocity = direction * speed
    move_and_slide()

func take_damage(amount):
    health -= amount
    if health <= 0:
        queue_free()  # 销毁自己（死亡）
```

如果你能看懂上面这段代码在说什么（角色移动、受伤、死亡），那你学 GDScript 就没什么障碍。看不懂也没关系——AI 会解释每一行。

## 三、语言选择：为什么不用 C#

Godot 支持两种语言：GDScript（自研）和 C#（微软的）。

| | GDScript | C# Godot |
|--|----------|----------|
| 学习曲线 | 像 Python，一行就能做事 | 需要学类/命名空间/类型声明 |
| 写一个角色移动 | ~8 行 | ~20 行 |
| AI 能帮你吗 | ✅ 很好 | ✅ 很好 |
| 性能 | 够用（2D 像素游戏没瓶颈） | 更高但用不到 |
| 与引擎集成 | 原生，所有教程都用它 | 需额外配置 .NET SDK |
| 适合场景 | 单人、快速开发、2D | 团队、复杂 3D、高性能需求 |

**结论：GDScript。** 你不是在做 3A 大作，2D 像素 RPG 的性能瓶颈在 GPU 渲染和图集管理上，不在脚本语言速度。GDScript 是 Godot 的"母语"，所有官方教程、社区例子、AI 训练数据都基于它。你用 C# 反而在英语国家找中文导游——能走，但绕路。

## 四、数据库和存储：从零到联机的演进

### MVP 阶段（现在）

```
本地 JSON 文件 + SQLite
```

- **JSON**：存玩家数据（等级、属性、物品栏、宠物）。一个文件就是一份存档。人眼可读，方便调试，AI 生成读写代码只需要 5 行。
- **SQLite**：一个文件就是一个数据库。当你需要查"背包里所有蓝色品质武器"这种结构化查询时，SQLite 比翻 JSON 快得多。Godot 有插件直接支持。

**为什么不用 MySQL/PostgreSQL 现在就搞**：你是单机游戏，不需要装额外的数据库服务。SQLite 是"文件即数据库"，零配置。等联机版时把数据迁移到 PostgreSQL，SQL 语法几乎一样，迁移成本低。

### 联机阶段（未来）

```
PostgreSQL + Redis
```

这是旧版设计文档提到的方案，方向正确。PostgreSQL 存所有持久数据（账号、角色、装备、摊位），Redis 存实时数据（在线玩家位置、战斗状态）。现在不需要装，知道有这个路线就行。

## 五、后端服务器：MVP 不碰，但架构要留位

**MVP 阶段**：没有后端。所有游戏逻辑运行在玩家本机。

**联机阶段**：需要后端来"仲裁"——验证玩家行为、同步世界状态。两个方案：

| 方案 | 优点 | 缺点 |
|------|------|------|
| **Nakama** | 游戏专用服务器，内置匹配/聊天/排行榜/存储，Docker 一键部署 | 学习曲线，Go 或 Lua 写逻辑 |
| **自建 Go 服务器** | 完全控制，轻量，性能好，Go 语言学习曲线平缓 | 聊天/匹配/排行榜都要手写 |

**推荐 Nakama**。你是个人开发者，Nakama 帮你解决了联机游戏 80% 的通用需求（账号系统、好友、聊天、匹配、排行榜），你只写"这个游戏的规则"（战斗怎么算、交易怎么处理）。自建的话这些基础功能你要手写半年。

但现在不说这个——等你 MVP 跑通、手感验证完，我们再决定联机架构。

## 八、版本管理：你第一天就要用

```
Git + GitHub
```

**为什么零基础也要从第一天就用 Git：**

你不写代码不会意识到：改了 10 个文件后游戏崩了，你不知道是哪次改动导致的。Git 可以让你**回退到"还没崩"的那个版本**，就像一个超级存档。

GitHub 是远程备份。你电脑坏了/重装系统了，代码在 GitHub 上不会丢。

你不需要学 Git 命令行。用 **GitHub Desktop**（图形界面，点按钮操作），日常操作就三步：
1. 写完一段功能 → 点 Commit
2. 写一句话描述改了啥
3. 点 Push 同步到云端

我帮你初始化仓库，之后每一步我都会告诉你。

## 十、地基：从现在就要做的架构习惯

虽然 MVP 是单机，但**从现在起按这个规矩写代码**，将来联机才不会重写：

### 规矩 1：游戏逻辑和画面显示必须分开

```
❌ 错误写法：角色攻击 → 直接扣怪物血条 → 播放受伤动画
✅ 正确写法：角色攻击 → 计算伤害值 → 通知"怪物"扣血 + 通知"画面"播放动画
```

游戏规则（伤害计算、等级检查、物品能否使用）放在独立脚本里，**永远不直接操作画面**。这样将来你把"游戏规则"搬到服务器上时，代码拿过去就能用。

### 规矩 2：所有数据用表驱动，不写死在代码里

```
❌ 错误写法：if monster_name == "红尾蝎": damage = 15
✅ 正确写法：读取 monsters.json，找到"红尾蝎"那行，取 damage 字段
```

JSON 文件就是你的"数值表"。加一个新怪物 = 在 JSON 里加一行，不用改代码。

### 规矩 3：存档用一个统一接口

不管本地存 JSON 还是未来服务器存 PostgreSQL，你调用 `SaveManager.save()` 的方式不变。底层实现可以换，上层代码不用动。
