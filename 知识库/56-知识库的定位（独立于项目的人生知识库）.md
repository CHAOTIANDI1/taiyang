## ⚠️ v3.1 知识库独立声明（凌驾性原则）

`知识库/` 是用户的**人生脑子**，记录用户人生所有知识——游戏设计、人生哲理、跨项目经验、读书笔记、行业洞察等。知识库**独立于《太阳之下》游戏项目**。详见 AGENTS.md §1.4 顶部声明 + §20.11 知识库技术栈独立性原则。

## 本实例

**遇到的问题**：
v3.0 之前（2026-07-19 及更早），知识库被定位为"《太阳之下》游戏的附属资料"——53 篇笔记全部围绕游戏设计展开，写作准则 v2.7 强制要求每篇笔记顶部加"本游戏实例"段。这一认知在 v3.1 升级时被用户推翻：

用户原话："知识库应该是我的一个总体人生的知识，而目前游戏的设计则是我人生项目的其中之一，但目前我们对知识库的看法变成了对这个游戏设计的知识而不是积累我的人生总项目知识。"

这句质疑暴露了知识库定位的根本错误——如果用户做下一个项目（Web/App/AI/读书笔记/人生哲理），知识库就废了一半。**正确的定位应该是：知识库是用户人生脑子，游戏项目只是其中一个章节。**

**专业名词/知识点**：
- **知识库定位**：知识库是用户人生脑子的总称，独立于任何具体项目
- **独立项目**：知识库本身是一个独立项目，不隶属任何游戏/Web/App 项目
- **用户人生脑子**：知识库的本质——用户一辈子的知识资产
- **跨项目复用**：做下一个项目时，知识库工具栈（tools/）原样搬走可用
- **凌驾性原则**：知识库独立性凌驾于游戏项目所有规则之上
- **本实例段（v3.1 扩展）**：原 v2.7 "本游戏实例"段扩展为"本实例"段——可以是本游戏实例/本人生实例/本读书实例/本跨项目实例，视笔记主题而定

**技术栈/代码**：
- AGENTS.md v3.1 顶部"⚠️ 知识库独立声明"
- AGENTS.md §0 项目身份表格 3 行：知识库定位 / 知识库技术栈 / 知识库工具目录
- AGENTS.md §1.4 顶部"⚠️ v3.1 知识库独立声明"
- AGENTS.md §20.11 知识库技术栈独立性原则
- docs/12-技术栈与架构.md "⚠️ v3.1 知识库独立声明（凌驾性原则）"段
- 本笔记（56 号）作为定位根本笔记

**应用过程**：
1. v3.0 落地后用户提出"知识库应该独立于游戏项目"
2. AI 复述用户立场：知识库是用户人生脑子，不是游戏附属资料
3. 走闸门 C 5 问差异审计：
   - 实际做法（独立）vs 原定（隶属游戏）哪里冲突？→ 定位根本性冲突
   - 原定为什么行不通？→ 锁死在单一项目，未来无法复用
   - 新做法是否更优？→ 是，跨项目复用 + 一辈子资产
   - 改规矩 vs 改做法代价？→ 改规矩（AGENTS.md 顶部加声明）影响面小
   - 是否同步回写？→ 必须回写 AGENTS.md + docs/12
4. AGENTS.md v3.1 升级：顶部加"⚠️ 知识库独立声明" + §0 加 3 行 + §1.4 顶部加声明
5. docs/12 同步加"⚠️ v3.1 知识库独立声明"段
6. 知识库工具栈从 GDScript 改为 Python（详见 [[54-Python工具脚本基础（命令行运行）]]）
7. v2.7 "本游戏实例"段扩展为"本实例"段（v3.1）
8. 本笔记（56 号）作为定位根本笔记

## 功能

### 知识库作为"用户人生脑子"的 4 个核心功能

1. **跨项目复用**：做下一个项目（Web/App/AI）时，整个知识库还能用，不用重建
2. **跨领域积累**：游戏设计 + 人生哲理 + 读书笔记 + 行业洞察 + 跨项目经验，统一管理
3. **独立技术栈**：Python 工具栈（`tools/`）不依赖任何具体项目，跨项目原样搬走
4. **用户一辈子的资产**：不会因为某个项目结束而失效，不会因为换工作而丢失

### 知识库 vs 游戏项目 docs/ 的区别

| 项 | 知识库（`知识库/`） | 游戏项目 docs（`docs/`） |
|----|--------------------|------------------------|
| 定位 | 用户人生脑子 | 游戏设计真值 |
| 范围 | 所有领域（游戏/人生/读书/跨项目） | 仅游戏设计 |
| 独立性 | 独立项目，跨项目复用 | 隶属《太阳之下》项目 |
| 技术栈 | Python（`tools/`） | GDScript（`scripts/`） |
| 终止条件 | 用户生命结束 | 游戏下架 / 项目结束 |
| 写作准则 | v2.7 8 段 + v3.1 扩展"本实例"段 | docs 自身的写作规范 |
| 谁来读 | 用户查询 + AI 辅助（不是 AI 必读） | AI 必读（设计真值） |

| 笔记主题 | 顶部段小标题 | 例子 |
|---------|-------------|------|
| 游戏设计相关 | `## 本游戏实例` | 53/54/55 号笔记（AI-RAG 5 层方案虽属知识库工具栈，但起源是游戏开发） |

**判断标准**：笔记内容如果只对游戏有用 → "本游戏实例"；如果对一辈子有用 → 选合适的"本 X 实例"。

## 原理

### 1. 为什么知识库独立于项目（5 条理由）

| # | 理由 | 解释 |
|---|------|------|
| 1 | 跨项目复用 | 未来做 Web/App/AI 项目时，整个知识库 + tools/ 原样搬走 |
| 2 | 跨领域积累 | 游戏设计 + 人生哲理 + 读书笔记 + 行业洞察统一管理 |
| 3 | 独立技术栈 | Python 工具栈不依赖任何具体项目 |
| 4 | 一辈子资产 | 不会因为某个项目结束而失效 |
| 5 | 凌驾性原则 | 知识库独立性凌驾于游戏项目所有规则之上 |

### 2. 凌驾性原则的本质

**凌驾性原则**意味着：当知识库与游戏项目发生冲突时，优先保护知识库的独立性。

| 冲突场景 | 凌驾性原则的判断 |
|---------|-----------------|
| 知识库工具用 GDScript（锁死 Godot） | ❌ 违反独立性 |
| 知识库工具放 `scripts/`（混入游戏代码） | ❌ 违反独立性 |
| 知识库引用游戏 `data/*.json` 字段 | ❌ 违反独立性 |
| 知识库只读 `scripts/` 文件元信息（不解析内容） | ✅ 符合独立性 |
| 知识库工具用 Python 写在 `tools/` | ✅ 符合独立性 |

### 3. 与游戏 docs/ 的本质区别

| 项 | docs/（游戏设计真值） | 知识库/（用户人生脑子） |
|----|----------------------|----------------------|
| 谁写 | AI 写 + 用户审 | AI 写 + 用户审 |
| 谁读 | AI 必读 | 用户查 + AI 辅助 |
| 内容 | 游戏该怎么设计 | 这是什么概念 + 为什么 |
| 终止 | 游戏下架 → docs 归档 | 用户生命结束 → 知识库留给后人 |
| 例子 | "怪物 HP=100" | "什么是 HP，为什么有 HP 这个概念" |

## 优势

### 1. 用户视角：一辈子的资产

- 学到的每个概念都记下来，不会忘
- 跨项目经验持续积累，越老越值钱
- 人生哲理沉淀，回头能翻
- 即使游戏结束了，知识库还在

### 2. AI 视角：跨项目一致的工具栈

- 下个项目 AI 进来，看到同样的 `tools/` 目录结构
- 同样的 `kb_inspector.py` + `sync_advisor.py` 可用
- 同样的 v2.7 准则 8 段写作标准
- AI 不用重新学习工具栈

### 3. 项目视角：游戏代码不动，知识库升级自由

- 知识库升级（如 v3.0 → v3.1）不影响游戏代码
- 游戏代码改动不影响知识库工具
- 两套技术栈（GDScript + Python）互不干扰
- 符合 §0.5 架构地基三问

## 使用场景

### 场景 1：写游戏设计笔记时

**准则**：用"本游戏实例"段。笔记内容围绕游戏设计展开。

**例子**：30-碰撞层与碰撞遮罩.md（讲碰撞层在《太阳之下》中怎么用）

### 场景 2：写知识库工具栈笔记时

**准则**：用"本实例"段。笔记内容是知识库工具的技术基础。

**例子**：54-Python工具脚本基础（命令行运行）.md

### 场景 3：写人生哲理笔记时

**准则**：用"本人生实例"段。笔记内容是用户的人生感悟。

**例子**：本笔记（56 号）+ 未来人生感悟类笔记

### 场景 4：写读书笔记时

**准则**：用"本读书实例"段。笔记内容是用户读某本书的感悟。

**例子**：未来读书笔记（如《代码整洁之道》读后感）

### 场景 5：未来做新项目时

**步骤**：
1. 整个 `tools/` 目录原样复制到新项目
2. 知识库笔记继续在新项目里写
3. 新项目的 `docs/` 是独立的（游戏/Web/App 各自的设计真值）
4. 知识库依然独立，记录跨项目经验

| 反例（违规） | 正例（合规） |
