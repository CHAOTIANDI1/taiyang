# 09 - 合规巡检机制

## 概念

每隔一段或换会话时，让 AI 主动跑一次"自查"——检查项目是否还遵守规矩。

生活类比：税务局定期来查账。不是不信你，是**定期被强制检查**自然合规。

## 功能

防止规矩定了一堆但没人执行——半年后项目失控了才发现代码里全是硬编码 `if name == ...`。

## 运作方式

### 5 项巡检

每次换会话或完成里程碑时，AI 必须主动运行一次：

```
1. 抽查最近 3 个 commit，文档是否同步更新
   → 看 commit 消息有没有"已同步 XX"
   → 看 docs/ 最近改动时间是否跟上 scripts/ 改动

2. 抽查 data/ 有没有新建未登记的 JSON
   → 对照 AGENTS.md §1.4 的 15 个 JSON 清单
   → 新文件不在清单 = 违规

3. 抽查 scripts/ 有没有硬编码 if name == ...
   → 全局搜 `if .* == ".*"` 模式
   → 发现一处就报警

4. 抽查 AGENTS.md §2.1 MVP 范围有没有越界
   → 比如做了后端代码、做了法师职业
   → 越界 = 违规

5. 抽查能力总目录（§1.8）有没有未登记的新系统
   → 比如偷偷新建了 dialogue_system 但没加进能力总目录
   → 新系统未登记 = 违规
```

### 输出格式

```
巡检报告：
❶ 三项一致：PASS / FAIL（原因）
❷ JSON 登记：PASS / FAIL（原因）
❸ 硬编码：PASS / FAIL（原因 + 文件:行号）
❹ MVP 范围：PASS / FAIL（原因）
❺ 能力总目录：PASS / FAIL（原因）
```

AI 不擅自修复违规——**只报告，等你决定**。

## 原理

**"持续集成"思想应用在文档**：与其等发现项目失控了再"n 个月修一次大补丁"，不如每周用 5 分钟小检查，小问题当场修。

类似：定期体检 vs 病了再去医院。前者可控成本，后者被动救火。

## 优势

| 优势 | 说明 |
|------|------|
| 防止规矩形同虚设 | 不检查的规矩等于没写 |
| 早发现问题 | 比如硬编码 5 处时容易修，50 处就难修了 |
| 责任分明 | AI 报告违规，你决定修不修 |
| 数据可追溯 | 每次巡检报告存档，看项目离规矩越来越远还是越来越近 |

## 使用场景

### 场景 1：完成一段任务关电脑前

打开对话跟 AI 说："跑一次合规巡检"。AI 跑完报告，你看 PASS 就收工。FAIL 的话你看原因决定下回修不修。

### 场景 2：隔了几天回来开新会话

按"对话开始协议"（方案 1），新 AI 第一件事是读 AGENTS.md + docs/00-总览.md + docs/12。**第二件事就跑合规巡检**，看上次离开后项目规矩有没有走偏。

### 场景 3：换 AI 模型

新模型进来跑一次巡检，**让新模型自己证明"我懂规矩了"**——通过 5 项检查报告，5 个 PASS = 它已掌握规则。有 FAIL 它先标出来再继续干活。
