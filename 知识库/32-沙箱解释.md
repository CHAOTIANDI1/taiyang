## 概念

**沙箱（Sandbox）** = 一种安全机制，把程序关在一个"密闭房间"里运行，房间里的所有读写操作都被限制在房间内，碰不到外面真实世界的文件系统。

**沙箱隔离区** = 沙箱给程序看到的"假象目录"。程序以为自己写到了 `D:\taiyang`，实际文件被悄悄写到了沙箱内部的另一个路径（比如 `C:\Users\xxx\AppData\...\work-mode-projects\xxxxx`）。

**路径重定向** = 沙箱拦截程序的写入请求，把目标路径从 `D:\taiyang\某文件` 偷偷换成 `沙箱内部路径\某文件`。程序感觉成功写了，但其实没写到真实位置。

生活比喻：
- **沙箱** = 玻璃温室。植物（程序）在温室里长得好好的，但根扎不进外面的土——所有水分养分都来自温室内部供给，碰不到外面真实的地。
- **沙箱隔离区** = 镜子迷宫。你在迷宫里看到前面有一扇门（你以为的 D 盘），走过去推开门其实只是推开了另一面镜子（沙箱内部路径）。
- **路径重定向** = 邮局转寄。你写信给"北京市 D 街 1 号"，邮局偷偷把信封地址改成"邮局内部代收箱 1234"，信到了但你以为寄到了原址。
- **沙箱透明读写** = 玻璃温室开了个对外的窗。窗内窗外同步——你浇水（写文件）外面真植物也能喝到水。

**反例 vs 正例**：

```
❌ 错误认知：AI 写文件到 D:\taiyang\xxx.md，文件就一定在物理 D 盘
   → TRAE Work + Work 模式：文件被重定向到沙箱隔离区
   → 用户在 D:\taiyang 看不到文件
   → AI 误以为"已写入"，用户误以为"AI 没干活"
   → 双方对项目状态认知割裂

✅ 正确认知：写文件后必须用"三方交叉验证"确认物理盘可见
   → Write 工具写文件 → RunCommand 用 Test-Path 查物理盘 → Read 工具回读内容
   → 三方一致才算真的写入物理盘
   → 任意一方不一致 = 沙箱在拦截
```

**本项目专属例子（v2.7 准则）**：

**例子 1：生动形象的好描述（帮理解概念本身）**
就像你在 `D:\taiyang` 写代码，TRAE Work 偷偷给你戴了副 VR 眼镜——你看到自己在 D 盘写代码，手也在动，但写的内容全部进了 VR 眼镜内部的虚拟 D 盘。你写完很满意，摘下眼镜发现物理 D 盘什么都没有。而 git push 就像你打电话让朋友去 GitHub 看你的代码——朋友通过 GitHub 网络看到的是你推上去的版本（沙箱里的），但物理 D 盘（你电脑上的）和 GitHub 上的不一致。这就是为什么 2026-07-18 那次 AI 以为改完了，用户却看不到。

**例子 2：本游戏实际应用（帮落地到本项目）**
2026-07-18 Phase 4.1 修复 5 个 bug 时，AI 在 Work 模式改了 `scripts/player/player.gd` 第 70 行 `_hitbox_area.collision_mask = 4`（修复 1：hitbox 排除自己），文件实际写到了沙箱隔离区 `C:\Users\27192\AppData\Roaming\TRAE SOLO CN\ModularData\ai-agent\work-mode-projects\6a55cfed862a92c0eaea46a7\scripts\player\player.gd`。git push 把代码推到 GitHub，但用户在物理 `D:\taiyang\scripts\player\player.gd` 看到的还是旧版本（没有 collision_mask=4）。这就是沙箱重定向导致的 §1.6 三项一致破坏——代码层（沙箱里）改了，但物理盘（用户看到的）没改，设计文档（docs/）和物理代码脱节。后来切换到 TRAE CLI 模式 + `git pull origin master` 才把 GitHub 的真值同步回物理盘。

## 功能

- **安全隔离**：防止 AI（或任何程序）误操作改坏系统关键文件（C:\Windows、注册表等）
- **权限控制**：只允许在指定目录内读写，越界请求被拒绝或重定向
- **环境复现**：每次运行环境干净，不受其他程序污染
- **副作用回收**：沙箱销毁后里面的修改全部消失（用于安全测试）

## 原理

### 为什么沙箱要拦截文件操作

操作系统的文件系统是全局共享的——任何程序都能读写任何路径（如果有权限）。AI 助手如果直接读写真实文件系统，存在风险：
- 误改系统关键文件（C:\Windows、注册表）
- 误删用户重要数据
- 恶意指令注入（用户让 AI 跑 `rm -rf /` 怎么办）

沙箱通过**虚拟化层**拦截所有文件 API 调用，把目标路径重映射到隔离目录，让程序在"假象"里自由操作，碰不到真实文件系统。

### 为什么 git push 能绕过沙箱

```
文件系统 API（Write/Edit/Read 工具）→ 沙箱拦截 → 路径重定向
网络协议（git push/pull）→ 沙箱不拦截网络 → 直接连 GitHub
```

git push 走 HTTPS/SSH 协议，是网络层操作，不是文件系统操作。沙箱主要拦截文件系统，对网络流量通常放行（否则程序连不上网）。所以：
- 沙箱里的代码修改 → 留在沙箱
- git commit + push → 推到真实 GitHub
- 结果：GitHub 远程仓库 ≠ 物理盘（这是 HANDOFF-20260718-1930 发现的核心矛盾）

### 为什么 TRAE CLI 模式沙箱透明

TRAE CLI 设计目标是"在用户指定的项目目录直接工作"，所以沙箱策略对项目目录子树开了白名单——所有对 `D:\taiyang\` 子树的读写直接透传到物理盘，不重定向。命令前缀虽然还带 `trae-sandbox '...'`，但这个包装只是统一执行入口，不改变文件落点。

## 使用场景

### 本项目踩过的坑（HANDOFF-20260718-1930 记录）

**现象**：TRAE Work + Work 模式下，AI 修复了 Phase 4.1 五个 bug，git push 到 GitHub 成功，但用户在物理 D:\taiyang 看不到任何修改。用户以为是 AI 没干活，AI 以为是用户没刷新。

**根因**：Work 模式沙箱把所有 D:\taiyang 的写入重定向到沙箱隔离区。git push 走网络绕过沙箱推到 GitHub，但物理盘没动。

**解决**：
1. 用户在物理 D 盘执行 `git pull origin master` 把 GitHub 最新同步到物理盘
2. 切换到 TRAE IDE 或 TRAE CLI 模式（沙箱对项目目录透明）
3. 用三方交叉验证法确认沙箱透明后再干活

### 三种模式选择建议

| 场景 | 推荐模式 | 理由 |
|------|---------|------|
| 长期开发本项目 | TRAE CLI | 沙箱透明，直接改物理盘，无割裂 |
| 临时切换/网页版 | TRAE IDE Code 模式 | 项目目录可写 |
| 实验/测试不希望污染环境 | TRAE Work Work 模式 | 沙箱隔离，销毁即回收 |
| 已经在 Work 模式写了东西 | 先 git push 再切 CLI 模式 git pull | 网络层绕过沙箱保数据 |

### 三方交叉验证法的复用场景

- 换新 AI 工具（如 Claude Code、Cursor）后首次写入前验证
- 怀疑文件没写到物理盘时排查
- 验证 .gitignore 是否真的忽略了某文件
- 验证 Read 工具看到的文件是否在物理盘（防止读到沙箱副本）

### 沙箱与本项目宪法的关系

- **AGENTS.md §1.6 三项一致原则**：沙箱隔离会导致"AI 改了代码但用户看不到"→ 三项一致被破坏。所以沙箱透明是三项一致的前提。
- **AGENTS.md §5.3 验证规矩**："不在没有验证证据的情况下声称完成"——沙箱模式下尤其需要三方交叉验证，否则会出现"假完成"。
- **sliver-vibe-coding Operating Law**："No fresh verification, no completion claim"——沙箱场景下，"fresh verification"必须包含物理盘可见性验证。

## 反例 vs 正例（常见错误）

### 错误 1：以为 Write 工具返回"成功"就等于物理盘可见
- **原因**：沙箱模式下 Write 成功只是写到了沙箱隔离区
- **修复**：用三方交叉验证法确认

### 错误 2：以为 git push 成功就等于物理盘已同步
- **原因**：git push 走网络绕过沙箱，推到 GitHub ≠ 物理盘已更新
- **修复**：物理盘单独执行 `git pull` 才能同步

### 错误 3：在 Work 模式干活后直接关掉沙箱
- **原因**：沙箱销毁后里面的修改全部消失
- **修复**：关闭沙箱前先 `git add . && git commit && git push` 把改动推到 GitHub

### 错误 4：以为 RunCommand 命令前缀的 `trae-sandbox` 等于沙箱在拦截
- **原因**：看到前缀就以为是 Work 模式沙箱
- **修复**：前缀只是统一执行入口，是否拦截要看三方交叉验证结果（CLI 模式下前缀在但透明）
