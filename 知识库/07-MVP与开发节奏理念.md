# 07 - MVP 与开发节奏理念

## 概念

**MVP = Minimum Viable Product**，最小可玩版本。先做出一个"能跑能玩的小东西"，验证核心手感是好的，再慢慢长成大游戏。

生活类比：先盖一室一厅样板间住进去试试——舒服了再扩成整栋楼。**不能一开始就盖 100 层空中楼阁**。

## 功能

把项目拆成"先做能跑的最小版本" → 跑通 → 验证 → 再扩展。**防止没做完就放弃**。

## 运作方式（举例）

### 反例：一次做全部

→ 花 6 个月闷头写所有功能 → 公测没人喜欢战斗手感 → 全部白做。

### 正例：先做 MVP

→ 2~4 周做 1 新手村 + 1 把剑 + 3 怪 + 1 Boss → 玩一下发现战斗好玩 → 再扩展。

具体 MVP 范围：
- 5 张地图（家+新手村+2 野外+小城镇）
- 1 把剑（剑士）
- 3~4 种怪 + 1 Boss
- 3 种宠物（拿到自动孵化）
- 2 个副职业（钓鱼+烹饪，顺序解锁）
- 1~15 级 + Lv15 转职
- 第一章完整剧情
- 邮件系统简化版

## 原理

**"延迟承诺"原则**：在你不知道玩家喜不喜欢当前手感之前，不做大的投入。先投小钱拿反馈，再决定要不要大投。

类似：餐厅新出菜品前先做小份给试吃，好评了再做全份上菜单。

## 优势

| 优势 | 说明 |
|------|------|
| 快速试错 | 2~4 周就能验证核心，不行就推翻重来 |
| 不会放弃 | 看到能玩的进度让自己热情保持 |
| 边做边学 | 前期你还在学 Godot，做小事正好对练 |
| 降低风险 | 资源集中投入在小核心上，先把"好玩"这个不确定因素消除 |

## 使用场景

### 场景 1：当前阶段

我们刚通过 16 份文档完整规划了"完整游戏"长什么样。现在不做完整游戏，先做 MVP。MVP 完成验证手感好之后，第 2 步做"缩小版全内容"（接入所有系统极简版）。

### 场景 2：MVP 跑通后

第 2 步（4~8 周）：做"缩小全内容"——接入宠物完整 AI、副职业 4 个、季节天气。
第 3 步：边做边讨论。

### 场景 3：联机版

> MVP → 缩小全内容 → 联机版

每个阶段都要 100% 通过该阶段验收清单（见 AGENTS.md §2.3）才能进下一阶段。
