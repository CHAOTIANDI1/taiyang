# 03 - 游戏规划核心概念

> 把项目里反复出现的核心设计/管理概念翻译成人话。

---

## MVP（最小可玩版本）

**生活类比**：先做个一室一厅的样板间住进去试试，舒服了再盖整栋楼。MVP 就是"样板间"。

**我们的 MVP**：1 个新手村 + 1 把剑 + 3 种怪 + 1 个 Boss + 钓鱼烹饪 + 1~15 级 + 第一章剧情。**做完验证手感 OK 才扩展**。

---

## 地基三规矩

这是 AGENTS.md §1 写的三条铁律，是整个项目防塌方的核心：

### 规矩 1：游戏逻辑与画面分离

**生活类比**：财务室（逻辑）和广告部（画面）分开。财务算完账把钱数告诉广告部，广告部再去做海报。不让广告部直接动钱。

**代码样例**：
```
✅ 对：attack_manager.gd → 算出伤害 50 → 通知 enemy 扣 50 血
                        → 通知 ui 显示 "-50" 飘字
❌ 错：attack_manager.gd → 直接改 enemy 的血条 + 直接种动画
```

**意义**：将来联机版把"逻辑"搬到服务器，"画面"还在玩家电脑。代码不用重写。

---

### 规矩 2：数据走 JSON，不写死代码

**生活类比**：饭店菜价贴在墙上，改价格换张纸就行。不要把价格刻在桌上——刻桌上每次改价要用刀子挖。

**代码样例**：
```
✅ 对：读 monsters.json → 红尾蝎 hp=50 → 用这个值
❌ 错：if name == "红尾蝎": hp = 50  （写死在代码里）
```

**意义**：调数值 = 改 JSON 一个数字。不用重新编译游戏。

---

### 规矩 3：存档走统一接口

**生活类比**：所有存的文件都进"档案室"。不管你是纸质文件（JSON）还是数字文件（数据库），都通过"档案管理员"统一存取，不能自己直接进档案室翻东西。

**代码样例**：
```
✅ 对：SaveManager.save(player_data)  SaveManager.load()
❌ 错：在每个系统里自己读写自己的存档文件
```

**意义**：单机版存 JSON，联机版换 PostgreSQL——上下层代码不用动，只换底层。

---

## 三项一致原则（7 条方案之 2）

**生活类比**：会计/出纳/银行必须三方对账。

任何一次提交必须三项同步：
| 项 | 是什么 |
|----|--------|
| ① 设计文档（docs/） | "这个功能该是什么样" |
| ② 数据文件（data/*.json） | "这个功能的数值" |
| ③ 代码（scripts/*.gd） | "这个功能的实现" |

改了一项必须同步另外两项（如果涉及）。没同步的提交**不算完成**。

---

## 能力总目录（7 条方案之 4）

**生活类比**：盖楼前贴在工地办公室的总平面图——今天只盖 5 楼，但看着图干活，知道哪堵墙不能拆（因为上面还要承重 20 楼）。

**是什么**：一张列了**游戏将来会有的所有系统**的表，每个标记当前状态（✅/❌/⏳），写明依赖关系。

**意义**：你想加新系统时先看表，有"❌"说明地基里留了位置；没列那就必须先加再动手。

---

## 7 条防塌方方案

| # | 名字 | 一句话 |
|---|------|--------|
| 1 | 对话开始协议 | 新 AI 会话第一动作必须是读宪法 + 复述项目状态 |
| 2 | 三项一致原则 | 文档/数据/代码必须同步，否则不算完成 |
| 3 | 数据文件清单 | 数字必须落在指定的 JSON 之一，不许散落 |
| 4 | 系统能力总目录 | 所有系统都列在一张表上，加新东西先查表 |
| 5 | 变更影响协议 | 引入新系统前必须答 4 问（提过没？影响谁？加哪个 JSON？三项一致？）|
| 6 | MVP 验收清单 | 4 大类必须 100% 打勾才算 MVP 完成 |
| 7 | 合规巡检 | 每完成里程碑/换会话时 AI 主动自查 5 项违规 |

## 这一节之后你会反复看到这些词

- **AGENTS.md**：项目宪法。所有 AI 必读。
- **docs/**：设计文档文件夹。
- **data/**：JSON 数据文件夹。
- **scripts/**：GDScript 代码文件夹。
- **MVP**：最小可玩版本。
- **P0/P1/P2/P3**：优先级排序。P0 必须做，P3 最靠后。
- **红线**：AGENTS.md §2 规定的"不许做"清单。
