# 08 - 三项一致原则

## 概念

任何一次代码提交必须保证三样东西同步：设计文档 / 数据文件 / 代码。缺少其中一样的提交不算完成。

生活类比：会计/出纳/银行三方对账。会计说 100，出纳说 100，银行说 100——三方一致才结账。

## 功能

防止"改了代码忘了改文档"——这是 AI 协作项目最常见的塌方原因。

## 运作方式

### 三项各是什么

| 项 | 放哪 | 是什么 |
|----|------|--------|
| ① 设计文档 | `docs/` | "这个功能该是什么样" |
| ② 数据文件 | `data/*.json` | "这个功能的数值" |
| ③ 代码 | `scripts/*.gd` | "这个功能的实现" |

### 反例：只改代码

```
AI 写完钓鱼系统的 .gd 脚本
→ commit 消息："完成钓鱼"
→ 但 fishing.json 没创建，docs/05-副职业.md 没更新
→ 下次 AI 读 docs/05 看不到钓鱼细节，自己重新发明
→ 项目失控
```

### 正例：三项同步

```
AI 完成钓鱼系统：
  1. 更新 docs/05-副职业.md 加钓鱼详细
  2. 新建 data/fishing.json 配置鱼品种
  3. 写 scripts/pet/fishing_system.gd 代码
  4. commit 消息："[新增] 钓鱼系统，已同步 05-副职业.md 和 fishing.json"
```

### 怎么落地

commit 消息必须写"[变更类型] 改了什么，文档/数据是否同步"。

变更类型有：
- `[新增]` ：加新功能
- `[修改]` ：改已有功能
- `[修复]` ：改 Bug
- `[文档]` ：只动文档
- `[数据]` ：只动 JSON

如果某次只改代码忘了文档，下次会话开始时补 commit："补：同步 05-副职业.md"。

## 原理

**"单一真值源"**（Single Source of Truth）：每个事实只有一处定义。
- 怎么玩 → docs/
- 多少数值 → data/
- 怎么实现 → scripts/

如果三个地方不一致，下一个人（或下一个 AI）不知道信谁，开始瞎写。

## 优势

| 优势 | 说明 |
|------|------|
| 防止文档过时 | 改功能必跟踪改文档 |
| 防止数据漂移 | 改功能必改对应 JSON |
| 换 AI 不丢规则 | 新会话读文档读到的是最新版 |
| 项目可审计 | 看 commit 知道是否同步 |

## 使用场景

### 场景 1：调整怪物血量

只改 data/monsters.json 不算功能变更，commit：`[数据] red_scorpion hp 50→80`。文档不变，代码也不变。

### 场景 2：加新武器

三项都要动：
- docs/06-物品体系.md 加武器说明
- data/equipment.json 加武器属性
- scripts/combat/ 导入武器模型（如需要）

commit：`[新增] 火焰剑，已同步 06-物品体系.md 和 equipment.json`

### 场景 3：改存档底层

.getJSONObject 改 PGSQL：
- docs/12 写上"联机版换 PostgreSQL"
- data/save_slots/ 保留 JSON（兼容存档）
- scripts/core/save_manager.gd 改底层

commit：`[修改] SaveManager 联机版底层切换 PostgreSQL`
