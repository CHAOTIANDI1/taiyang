# 23 - 规则执行滑坡根因与 3 道防坡闸门

## 概念

**规则执行滑坡** = 规则写在文档里，AI 实际工作时不严格执行甚至跳过。

生活类比：法律存在但执行不到位——有红绿灯但司机闯黄灯。规则在场 ≠ 规则被执行。

## 功能

防止 AI"接到任务就冲，凭记忆工作，漏掉应当同步触发的连带规矩"。本项目用户对滑坡**零容忍**。

## 运作方式

### 根因：为什么 AI 会滑坡

| 触发 | 后果 |
|------|------|
| 接到紧迫任务 | AI 把目标聚焦在"完成任务本身"，连带规矩被挤到次要 |
| 没有执行前清单 | AI 凭"我记得规矩有哪些"工作，必漏 |
| 没有执行后核对 | AI 不知道自己漏了什么，自以为完成 |
| 实际做法与原定冲突时 | AI 容易擅自调整，破坏地基 |

### 3 道闸门

#### 闸门 A：任务前"触发清单核对"

任务开始前 AI 必须打印：

```
任务：<描述>
适用规矩：
  - §X.X 内容 → 需要做什么
触发条件核对：
  - §1.4 知识库补全第 9 项？
  - §1.6 三项一致？
  - §6.3 合规巡检？
确认完毕，开始执行。
```

类比：消防员出动前念"7 项出动清单"——念完才上车。

#### 闸门 B：任务后"完成审计 7 问"

任务完成后 AI 必须答：

1. 改了 code 同步改 docs 了吗？
2. 改了 JSON 同步改 code 了吗？
3. AGENTS §2.1 MVP 范围有没有越界？
4. §1.8 能力总目录有没有未登记的新系统？
5. §1.4 知识库有没有该补未补的笔记？
6. 三项一致性验证通过吗？
7. commit 信息按 §1.6 格式吗？

7 项全 ✅ 才算完成。

类比：飞行员降落前念"着陆检查单"全过才落地。

#### 闸门 C：差异审计 5 问

实际做法与原定不一致时 AI 必答：

1. 实际做法与原定哪里冲突？
2. 原定为什么行不通？
3. 新做法是否更优？证据是什么？
4. 改规矩 vs 改做法，代价哪个小？
5. 是否同步回写 docs/ 或 AGENTS.md？

只有 5 问答完且用户确认后才能变更。事后必须回写真值。

类比：施工发现图纸行不通，先填"工程变更联系单"5 问请监理签字再动。

## 原理

AI 的工作流必须有"**前后端强制控制流**"：
- 前端：执行前列清单——不知道哪些规矩该用就念一遍
- 后端：执行后自检——做完每一项 × 检查 7 问
- 异常：做法与原定冲突时走变更协议，不擅自改

缺任一端都漏。本宪法 v2.5 把这 3 端全部硬编码进 §1.4 后。

## 优势

| 优势 | 说明 |
|------|------|
| 规矩必执行 | 任务前核对清单，不靠记忆 |
| 防半完成 | 7 问全过才算完成 |
| 防擅自变地基 | 5 问答完用户确认才动 |
| 历史可追 | Git commit 看到每次是否走流程 |
| 治本 | 不靠"AI 自觉"靠"流程强制" |

## 使用场景

### 场景 1: 修一个报错

闸门 A 打印：触发 §1.4 第 9 项报错三步流程（解释根因+修+补知识库）
→ 修复
→ 闸门 B 7 问：知识库补了？文档同步？commit 格式？
→ 全过算完成

### 场景 2: 想换一种代码架构

闸门 C 5 问：为什么原定架构行不通？新架构证据是什么？影响面？
→ 用户确认
→ 同步回写 docs/12-技术栈与架构.md
→ 走三项一致 commit

### 场景 3: 完成一个 Phase

闸门 B 7 问跑 + §6.3 合规巡检跑
→ 7 问全过 + 5 项巡检 PASS 才算 Phase 完成

## 与现有规矩关系

- 闸门 A 服务 §1.5 对话开始协议 + §1.4 知识库运维
- 闸门 B 服务 §1.6 三项一致 + §6.3 合规巡检
- 闸门 C 服务 §15 新技术引用审计 + §16 修改本文件

**没破坏任何现有规矩，是把现有规矩强制执行的"程序外壳"**。
