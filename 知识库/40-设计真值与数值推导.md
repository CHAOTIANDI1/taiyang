## 概念

**设计真值（Source of Truth）** = 一个游戏设计决策的"权威答案"只存在于一个地方，其他地方都是它的"副本"或"推导值"，不能自己说了算。

**生活比喻 1（生动形象）**：一个公司只有一个老板，所有决策都听老板的。如果张三问老板"今天加不加班"，老板说"加"，那李四再问张三时，张三也得说"加"——不能因为李四是新人就告诉他"不加"。老板的话就是"设计真值"，张三李四传话不能改。

**生活比喻 2（本游戏应用）**：《太阳之下》的传送兜底距离，docs/03 第 100 行说"15 身位"。这就是设计真值。代码 pet.gd 写 `TELEPORT_DISTANCE = 480.0` 时，480 不是凭空想的，是从 docs 推导出来的：15 × 32 = 480。如果代码写 300，就是"代码改了真值"——违反原则。

**反例 vs 正例**：
```
❌ 错误（代码自己定数值）：
# pet.gd
const TELEPORT_DISTANCE: float = 300.0  # 拍脑袋
# docs/03 说 15 身位 = 480，但代码写了 300，两边对不上
# 用户玩游戏发现传送太频繁，改 docs 还是改代码？混乱

✅ 正确（docs 是真值，代码推导）：
# docs/03 第 100 行
"宠物与玩家距离超过 15 个身位且 3 秒内未缩短，直接传送到玩家身边"
# pet.gd
const TELEPORT_DISTANCE: float = 480.0  # 15 身位 × 32 像素 = 480，推导自 docs
# 两边对应，用户问"为什么是 480"，AI 答"15 身位 × 32 像素推导"，有据可查
```

## 功能

- **真值唯一**：每个设计决策只有一个权威来源（docs/）
- **数值推导链**：docs 写"15 身位" → 代码计算"15 × 32 = 480 像素"
- **多源头冲突检测**：代码/数据/文档对不上时，走闸门 C 差异审计 5 问
- **追溯性**：任何数值都能回溯到 docs 哪一行，不是拍脑袋

## 运作方式

### 4.4.5 传送兜底的真值推导过程

**第一步：找 docs 真值**

打开 `docs/03-宠物系统.md` 第 100 行：
```
- **防卡死机制**：宠物与玩家距离超过 15 个身位且 3 秒内未缩短，直接传送到玩家身边
```

**真值**：15 身位 + 3 秒

**第二步：推导像素值**

| 概念 | 数值 | 来源 |
|------|------|------|
| 1 身位 | 32 像素 | 角色统一 32×32 像素（AGENTS.md §0 项目身份） |
| 15 身位 | 15 × 32 = 480 像素 | 推导 |
| 3 秒 | 3.0 秒 | docs 直接写 |

**第三步：代码落地**

```gdscript
# pet.gd 第 28-30 行
# 4.4.5 传送兜底常量（防卡死，符合 docs/03 设计：15 身位 × 32 像素 = 480）
const TELEPORT_DISTANCE: float = 480.0  # 15 身位，距离超过此值启动计时器
const TELEPORT_TIMEOUT: float = 3.0     # 3 秒内距离未缩短就传送
```

**关键**：注释里写"15 身位 × 32 像素 = 480"，让任何人看到这行代码都能回溯到 docs。

### 代码逐行执行流程

**场景：玩家跑远，宠物追不上**

```
帧 1：玩家在 (100, 100)，宠物在 (200, 100)
  distance = 100 像素 < 480 → 不触发

帧 100：玩家跑到了 (1000, 100)，宠物还在 (300, 100)
  distance = 700 像素 > 480 → _teleport_timer 开始累计

帧 100-280（3 秒约 180 帧）：玩家继续跑，宠物追不上
  distance 一直 > 480
  _teleport_timer 累计到 3.0 秒

帧 281：_teleport_timer >= 3.0 → 触发传送
  宠物位置 = 玩家位置 + (80, 0)  # 右侧 80 像素（FOLLOW_OFFSET）
  打印：[Pet:pet_001_ham_dog] 传送兜底！距离=720.5 > 480 像素（15 身位），3 秒未缩短，传送到玩家身边
```

## 原理

### 为什么要有"设计真值"概念？

**问题**：游戏里同一个设计决策可能出现在多个地方——
- docs/03 写"15 身位"
- 代码 pet.gd 写 480
- 数据 pets.json 可能写"teleport_distance": 480
- 测试用例可能写 500

如果没有"真值唯一"原则，4 个地方各有各的数，改一处忘改其他 3 处，bug 就出来了。

**解决**：AGENTS.md §3.1 规定 docs/ 是真值唯一来源。代码/数据是推导值，必须能追溯。

### "身位"为什么是 32 像素？

AGENTS.md §0 项目身份：
```
| 像素密度 | 32×32（角色/怪物/NPC 统一）|
```

角色 32×32 像素 = 1 个身位。这是项目级常量，所有"X 身位"的描述都按 32 像素推导。

### 真值冲突时怎么办？

走闸门 C 差异审计 5 问（AGENTS.md v2.5）：
1. 实际做法与原定哪里冲突？
2. 原定为什么行不通？
3. 新做法是否更优？证据是什么？
4. 改规矩 vs 改做法，代价哪个小？
5. 是否同步回写 docs/？

**只有 5 问答完且用户明确确认后，才能变更既定规矩或做法**。

## 优势

| 优势 | 说明 |
|------|------|
| 一致性 | docs/代码/数据/测试全对应，不会出现"代码说 300 docs 说 480"的混乱 |
| 可追溯 | 任何数值都能回溯到 docs 哪一行，不是拍脑袋 |
| 防漂移 | 改一处忘改其他时，闸门 C 检测出来 |
| 沟通清晰 | 用户问"为什么是 480"，AI 能答"15 身位 × 32 像素推导" |
| 维护容易 | 想改数值，只改 docs 真值，代码/数据自动跟着改 |
