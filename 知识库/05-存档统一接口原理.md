# 05 - 存档统一接口原理

## 概念

所有存档操作必须走 `SaveManager` 这个"档案管理员"，不能每个系统自己读写存档文件。

## 功能

将来单机换 PostgreSQL（联机版），只要换底层——上层代码全部不用动。

## 运作方式（举例）

### 反例：各系统自己存

```gdscript
# player.gd
func save_self():
    File.write("save1.json", hp)        # 直接写文件

# inventory.gd
func save_self():
    File.write("save2.json", items)     # 直接写另一个文件

# pet.gd
func save_self():
    File.write("save3.json", pet_data)  # 再一个文件
```

问题：
- 存档散在多个文件，换数据库要改 3 处
- 存档槽位没法统一管理（3 个系统可能存到不同槽）
- 一个系统存失败另一个不知道，数据不一致

### 正例：统一接口

```gdscript
# scripts/core/save_manager.gd（管家）
func save(slot_name, data):
    # 实际写到 data/save_slots/<slot_name>.json
    write_file(...)

func load(slot_name) -> Dictionary:
    # 读 data/save_slots/<slot_name>.json
    return data

func delete(slot_name):
    remove_file(...)
```

调用方只管调管家：
```gdscript
# player.gd
SaveManager.save("slot1", {"hp": 100, "mp": 30})

# inventory.gd
SaveManager.save("slot1", {"items": [...]})

# pet.gd
SaveManager.save("slot1", {"pet": {...}})
```

或者更高级的方式：
```gdscript
# 一键打包所有系统的状态
SaveManager.save_game_state("slot1")  # 内部协调所有子系统
```

## 原理

底层是"接口隔离"（Interface Segregation）：
- 上层调 `SaveManager.save()`，**不知道也不关心底层是 JSON 还是 PostgreSQL**
- 底层可换：JSON → SQLite → PostgreSQL → Redis Cache + PostgreSQL

类比：银行存取款的窗口只问"存多少 取多少"，不关心你用什么货币存款账号是什么卡——窗口是统一接口。

## 优势

| 优势 | 说明 |
|------|------|
| 切换数据库无痛 | 单机换联机改底层一行代码 |
| 多系统存档一致 | 全在一个槽里，一次 commit |
| 存档槽位好管理 | 3 手动 + 1 自动，SaveManager 控制不冲突 |
| 失败可回滚 | 比如自动存档失败可恢复上一份 |

## 使用场景

### 场景：MVP 单机阶段

SaveManager 底层实现 = 写 `data/save_slots/slot1.json` 文件，里面是 JSON 字符串。

玩家进入新地图 → 游戏调 `SaveManager.save("auto", current_state)` → 写自动槽。

玩家手动存档 → UI 调 `SaveManager.save("manual_01", current_state)` → 写第 1 手动槽。

### 场景：联机版阶段

SaveManager 底层实现改成：调服务器 API `/save-state`，服务器写 PostgreSQL。
**上层代码完全不变**——还是 `SaveManager.save("auto", ...)`。

### 场景：将来加新系统

比如加一个公会系统，要存公会数据：
- 旧方式：公会自己写文件 → 要先想清楚怎么跟其他系统的存档合并
- 新方式：调 `SaveManager.save("slot1", guild_data)` → 自动合入主存档
