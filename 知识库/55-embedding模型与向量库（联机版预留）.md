## 本游戏实例（v2.7 强制段）

**遇到的问题**：
知识库 AI-RAG 5 层方案中，第 1 层（语义索引）和第 4 层（跨笔记推理）被延后到联机版。用户问"为什么延后？现在不能做么？"——这暴露出用户第一次见 embedding、向量库、BGE-M3、SQLite-vec 这些词，需要生活比喻解释清楚。

**专业名词/知识点**：
- **embedding 模型**：把文字变成一串数字（向量）的 AI 模型，让 AI 能"算"语义相似度
- **向量（Vector）**：一串数字（如 [0.12, -0.45, 0.78, ...]），代表文本的"语义指纹"
- **向量库（Vector Store）**：专门存储和快速查询向量的数据库
- **语义搜索**：按"意思"找内容，而不是按"字面"找
- **BGE-M3**：开源的 embedding 模型，可本地运行，免费
- **SQLite-vec**：轻量级向量库，基于 SQLite 扩展，文件即数据库
- **关键词搜索（Ctrl+F）**：按字面匹配，搜"狗"只能找到"狗"字

**技术栈/代码**：
- MVP 阶段：不引入（用 Ctrl+F 替代）
- 联机版阶段：BGE-M3 + SQLite-vec（待选型）
- 触发条件：笔记破 200 篇时

**应用过程**：

## 概念

### 1. 生动形象例子（图书馆的智能检索机）

把知识库想象成一座图书馆，53 篇笔记就是 53 本书。

**普通检索（Ctrl+F）= 死磕字面**：
- 您要找"狗"，检索机只能在书名和正文里找"狗"字
- 一本讲"宠物训练"的书里没有"狗"字 → 找不到
- 但这本书明明讲的就是狗 → 检索机漏了

**语义检索（embedding）= 理解意思**：
- 检索机事先给每本书做"内容指纹"（一串数字）
- 您搜"狗"，检索机把"狗"也变成指纹
- 然后算"狗的指纹"和"每本书的指纹"之间的距离（相似度）
- "宠物训练"的指纹和"狗"的指纹距离近 → 推荐给您

**关键差异**：
| 检索方式 | 搜"狗"能找到 |
|---------|------------|
| Ctrl+F | "狗"、"小狗"、"狗粮" |
| 语义检索 | "狗"、"小狗"、"宠物"、"汪星人"、"犬类" |

## 功能

### embedding 模型的 3 个功能

1. **文字转向量**：输入一段文字，输出一串数字（通常是 768 维或 1024 维的向量）
2. **保留语义**：相似意思的文字，向量距离近；不同意思的，向量距离远
3. **跨语言**：高级模型支持多语言（中文"狗"和英文"dog"向量相近）

### 向量库的 2 个功能

1. **存储向量**：把所有笔记的向量存起来，下次不用重新算
2. **快速查询**：给一个查询向量，快速找到最相似的 K 个向量（KNN 算法）

### 两者配合的完整流程

```
建库阶段（一次性）：
  for 每篇笔记:
    向量 = embedding_model(笔记内容)
    vector_store.save(笔记ID, 向量)

查询阶段（每次问问题）：
  查询向量 = embedding_model(用户问题)
  相似笔记 = vector_store.search(查询向量, top_k=3)
  AI 读相似笔记 + 生成答案
```

## 原理

### 1. embedding 模型怎么训练的

模型在大量文本上训练，学习"出现在相似上下文的词，语义相近"。

例如：
- "我喜欢养**狗**" 和 "我喜欢养**宠物**" → 上下文相同 → 狗和宠物的向量靠近
- "狗" 和 "汽车" → 上下文很少重叠 → 向量距离远

训练结果：模型学会把"语义相近的文本"映射到"向量空间相近的位置"。

### 2. 向量相似度怎么算

最常用的是**余弦相似度**（Cosine Similarity）：

```
两个向量 A 和 B 的余弦相似度 = (A · B) / (|A| × |B|)
取值范围：[-1, 1]，越接近 1 越相似
```

生活比喻：两个向量像两根指针，余弦相似度就是"两根指针夹角的余弦值"。夹角越小（方向越一致），相似度越高。

### 3. 为什么 MVP 不做

| 理由 | 详细说明 |
|------|---------|
| 笔记太少 | 当前 53 篇，Ctrl+F 1 秒能搜完，引入收益小 |
| 装环境成本 | 要装 Python + sentence-transformers + 下载 2GB 模型 |
| 违反 §19 过早抽象 | 没有用户反馈说 Ctrl+F 不够用，凭啥判断需要语义搜索？ |
| 不符合 §0.5 第二问 | 加新系统（装 Python）时要重新配置环境，地基不够稳 |

### 4. 为什么联机版做

| 理由 | 详细说明 |
|------|---------|
| 笔记变多 | 联机版启动后笔记可能破 200 篇，Ctrl+F 真的不够用 |
| 用户反馈触发 | 那时用户会真实反馈"搜不到"，有具体执行依据 |
| 一次性成本可摊薄 | 装一次环境，用 5 年，性价比高 |
| 符合 §0.5 第三问 | 加 BGE-M3 不影响游戏代码（知识库是辅助层） |

## 优势

### 1. 能搜到"意思相近"的内容

Ctrl+F 搜"狗"找不到"宠物"，embedding 能找到。

### 2. 跨语言搜索

中文问题能匹配英文笔记（如果有的话）。

### 3. AI 不用读全部笔记

```
传统 AI 问答：AI 要读 200 篇笔记 → 上下文爆炸 → 答不准
embedding + AI 问答：AI 只读 top 3 相似笔记 → 上下文精简 → 答得准
```

### 4. 发现隐含关联

第 4 层跨笔记推理：embedding 能发现"49 号笔记"和"27 号笔记"虽然讲不同主题但本质同模式，建议加双链。

## 使用场景（含对接）

### 场景 1：联机版知识库语义搜索（第 1 层）

**对接**：
- 输入：用户问题（自然语言）
- 处理：BGE-M3 转向量 → SQLite-vec 查 top 5 → AI 读 + 生成答案
- 输出：4 段格式答案（直接答案 + 出处 + 反例 vs 正例 + 关联）

### 场景 2：联机版跨笔记推理（第 4 层）

**对接**：
- 输入：所有笔记向量
- 处理：聚类分析 → 找语义相近但无双链的笔记对
- 输出：报告"建议给笔记 X 和笔记 Y 加双链"

### 场景 3：永远不会用到的场景

- ❌ 游戏内 NPC 对话搜索（NPC 对话走 dialogues.json，不需要语义搜索）
- ❌ 怪物 AI 决策（怪物 AI 走 monsters.json + 状态机，不需要向量）
- ❌ 玩家背包搜索（背包走 InventoryManager + ID 匹配，不需要向量）

**关键**：embedding 是知识库专用，不进入游戏运行栈。

## 反例 vs 正例

### 反例 1：MVP 阶段就引入 embedding

```
❌ 现在（53 篇笔记）就装 BGE-M3 + SQLite-vec
```

**问题**：
- 装 Python + 下载 2GB 模型，零基础用户大门槛
- 53 篇笔记 Ctrl+F 1 秒能搜完，引入收益小
- 违反 §19 过早抽象
- 一旦引入就要维护，加重项目栈

### 正例 1：MVP 用 Ctrl+F + 第 2/3/5 层

```
✅ 现在：Ctrl+F + AI 问答 + 自动巡检 + 项目同步
   未来：联机版笔记破 200 篇时再引入 embedding
```

**优势**：
- 当前 53 篇够用
- 未来引入时不影响现有架构
- 符合 §0.5 三问

### 反例 2：用 OpenAI 云端 API 做 embedding

```
❌ embeddings.create(model="text-embedding-3-small", input="...")
```

**问题**：
- 要 API Key（用户没有）
- 持续付费（按调用次数计费）
- 要网络（离线不能用）
- 隐私问题（笔记内容上传到 OpenAI）

### 正例 2：用 BGE-M3 本地模型

```
✅ SentenceTransformer('BAAI/bge-m3')
```

**优势**：
- 免费开源
- 本地运行，不要网络
- 隐私安全（笔记不出本机）
- 一次性下载，永久使用

### 反例 3：把向量索引进 git

```
❌ git add 知识库/.ai-index/vectors.db
```

**问题**：
- vectors.db 是派生数据（从 .md 重建）
- 进 git 会污染 history（每次重建都改）
- 文件可能几 MB，仓库膨胀

### 正例 3：派生数据 .gitignore 排除

```
✅ .gitignore 加规则：
   知识库/.ai-index/
   知识库/.ai-reports/
```

**优势**：
- 仓库干净
- 可重建（删了再跑一次建索引脚本即可）
