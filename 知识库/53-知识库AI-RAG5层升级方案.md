## 概念

### 1. 生动形象例子（图书馆升级）

把知识库想象成一座图书馆，5 层升级就像 5 次升级改造：

| 层 | 改造内容 | 类比 |
|---|---------|------|
| 第 1 层 语义索引 | 给每本书做"内容指纹" | 图书馆装"智能检索机"，按内容找书而非按书名 |
| 第 2 层 AI 问答 | 配个图书管理员 | 你问"宠物怎么跟随"，管理员直接答 + 拿书给你看 |
| 第 3 层 自动巡检 | 每晚扫一遍书架 | 找出断链接、重复书、过时书、不达标书 |
| 第 4 层 跨笔记推理 | 管理员发现隐含关联 | "49 号和 27 号讲的是同一种模式，要不要加个参见？" |
| 第 5 层 项目同步 | 盯着代码仓库 | "代码 X 改了，笔记 Y 还没动，要不要回填？" |

**关键差异**：传统 AI-RAG 让 AI 自由生长，我们的 AI 是"图书馆员 + 守门员"——能帮你找书、能建议补书，但不能擅自改书（必须走闸门 C 用户确认）。

### 2. 本游戏实际应用例子

用户问"红尾蝎的掉落机制是怎么设计的？"：

```
传统 AI-RAG（无闸门）：
  AI 答："根据红尾蝎的 drops 数组，40% 概率掉落 material_leather_001..."
  → 用户不知道答案来自哪里
  → AI 可能瞎编
  → 没有出处可验证

我们的 AI（AI + 闸门）：
  AI 答（4 段格式）：
    1. 直接答案：红尾蝎 40% 概率掉落普通皮革，由 monsters.json 的 drops 数组定义
    2. 出处标注：
       - 笔记：[[52-满包入邮件与怪物掉落与InventoryUI]] 第 7 段
       - 代码：scripts/core/loot_helper.gd 的 drop_loot() 函数
       - 数据：data/monsters.json 第 48 行 drops 数组
    3. 反例 vs 正例：
       - ❌ 反例：硬编码 if monster_id == "red_scorpion": drop leather
       - ✅ 正例：数据驱动 drops 数组 + LootHelper.drop_loot 通用函数
    4. 关联推荐：[[49-物品体系与数据驱动]] / [[51-EquipmentManager装备耐久]]
  → 如果知识库没记载：主动说"建议走闸门 A 触发补全"
  → 如果需要新建笔记：走闸门 C 差异审计 + 用户确认后才写
```

| 巡检项 | 触发条件 | 严重度 | 自动/人工 |
|--------|---------|:------:|:--------:|

## 原理

### 1. 为什么不引入 AI 自动写入

| 方案 | 优点 | 缺点 |
|------|------|------|
| AI 自动写入笔记 | 高效，省用户决策 | 质量失控，可能产生垃圾笔记，违反 §6.1 安全规矩 |
| AI 提建议 + 用户确认 | 质量可控，用户决策权 | 慢一点，但符合"用户对历史零容忍" |

选了"AI 提建议 + 用户确认"：
- 符合 §6.1 "不删除 Git 历史"的精神（用户对历史零容忍）
- 符合 §7 决策权限："游戏设计变更由用户决定"
- 防止 AI 滑坡——AI 不能擅自改知识库

### 2. 为什么 v3.1 不做第 1 层和第 4 层（v3.2 已推翻此判断）

> ⚠️ **v3.2 修订**：本段是 v3.1 原判断，v3.2 已推翻。保留此段作为滑坡反思的证据——v3.1 用"游戏 MVP 视角"判断知识库决策，是教科书级别的滑坡。详见 [[57-AI滑坡识别（游戏MVP与知识库独立性的边界混淆）]]。

| 层 | v3.1 不做的理由（已推翻） | v3.2 改为本期做的理由 |
|---|----------|--------------|
| 第 1 层 语义索引 | 需要 embedding 模型 + 向量库，违反 §19 过早抽象 | 知识库是用户人生脑子（§20.11 凌驾性原则），用"一辈子最优"标准判断，BGE-M3 2GB 在一辈子尺度微不足道 |
| 第 4 层 跨笔记推理 | 依赖第 1 层索引 | 复用第 1 层索引，技术栈统一，无需引入新技术 |

**v3.1 核心判断（已推翻）**：用户指出"知识库是辅助层，不影响游戏运行栈"——这个判断对，但 v3.1 错误地延伸为"必须立即引入所有层"。第 1/4 层需要新技术栈（embedding 模型），引入成本高，当前 52 篇笔记用第 2/3/5 层够用。

**v3.2 推翻理由**：v3.1 的判断违反 §20.11 凌驾性原则——用"游戏 MVP 57 篇笔记够用"判断"知识库一辈子资产的地基"是跨域挂钩。知识库独立视角下，BGE-M3 是 SOTA 中文语义 embedding，2GB 在一辈子尺度微不足道，用户硬件 4060 够用，应该选最优而非轻量。

### 3. AI + 闸门协同的 4 个价值

```
传统 AI-RAG：
  用户问 → AI 答 → AI 自动生成笔记 → 知识库膨胀 → 质量失控

我们的 AI + 闸门：
  用户问 → AI 答（4 段格式，标注出处）
       → AI 建议补笔记（不自动写）
       → 走闸门 A 打印清单
       → 走闸门 C 差异审计（如果是大改）
       → 用户确认
       → AI 写笔记（按 v2.7 准则 8 段）
       → 走闸门 B 第 8 问验收（准则符合度）
       → 入库 + 更新索引
       → 走第 3 层巡检（30 天后复查）
```

4 个价值：
1. **AI 提效率，闸门保质量**——AI 写得快，闸门拦得严
2. **AI 提建议，用户做决策**——AI 不能擅自改知识库
3. **AI 做巡检，闸门做验收**——AI 主动发现问题，闸门强制修复
4. **AI 跨会话记忆，闸门防遗忘**——AI 用索引找回记忆，闸门确保每个会话都走协议

### 4. 派生数据不进 git 的理由

- `知识库/.ai-index/` 向量索引：可从 .md 文件重建，进 git 会污染 history
- `知识库/.ai-reports/巡检-*.md` 巡检报告：可重新生成，历史报告无价值
- `knowledge_backup/` 本地备份：临时文件，不进 git

**判断标准**：派生数据（可从源头重建）不进 git，源头数据（人工写的 .md 笔记）进 git。

## 优势

1. **不破坏现有治理框架**：AGENTS.md 闸门机制不变，AI 是闸门的执行者
2. **分阶段引入**：5 层按需启用，不一次性引入所有新技术
3. **用户决策权保留**：AI 不能擅自改知识库
4. **质量可控**：4 段格式 + 5 项巡检 + v2.7 准则三重保险
5. **跨会话连贯**：报告归档 + 巡检机制让 AI 在新会话能恢复状态
6. **不影响游戏运行栈**：知识库升级是辅助层，不动 §1.1/§1.2/§1.3 三条地基
