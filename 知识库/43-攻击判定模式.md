# 43 - 攻击判定模式

## 本游戏实例（v2.7 准则）

- **遇到的问题**：
  - **Phase 4.4.6 步骤 2**：怪物前摇预警结束后**必中**——玩家即使跑开了也挨打
  - 这种"必中"机制让战斗没有躲避空间，玩家会觉得"我跑了还挨打不公平"
  - 需要让怪物攻击有"必中"和"可躲"两种模式：必中给 Boss 大招，可躲给普通怪普攻
- **专业名词/知识点**：攻击判定模式（attack_pattern）、前摇后范围检查（telegraph_then_check）、必中（guaranteed_hit）、瞬时检测（instant_check）、miss 落空、attack_range 攻击距离
- **技术栈/代码/美术**：GDScript `match` 语句、`monsters.json` 的 `attack_pattern` 字段、`pets.json` 的 `attack_pattern` 字段、`monster.gd._do_attack()`、`monster.gd._apply_damage_to_target()`
- **应用过程**：
  1. 在 `monsters.json` 给 red_scorpion 加 `"attack_pattern": "telegraph_then_check"` 字段
  2. 在 `monster.gd._load_data_from_json()` 读字段到 `_attack_pattern` 变量
  3. 改造 `_do_attack()`：用 `match _attack_pattern` 分支处理
     - `guaranteed_hit` → 直接命中（Boss 用）
     - `telegraph_then_check` → 检查目标距离 ≤ attack_range → 命中 / 否则 miss 飘字
  4. 抽出 `_apply_damage_to_target()` 函数（命中并扣血）供各分支复用
  5. 给 `pets.json` 加 `"attack_pattern": "instant_check"` 字段保持一致性（宠物无预警）

---

## 概念

**攻击判定模式**是"怪物前摇结束后怎么判断打没打中"的规则。同样是前摇 0.5 秒，结束后的判定方式可以不同：必中、可躲、瞬时检测。

**生活比喻 1（生动形象）**：拳击比赛的三种计分方式：
- **必中模式**：裁判数完秒就自动计分，选手打不打到都算（Boss 大招，不可躲）
- **前摇后检查**：裁判数完秒后看选手有没有真的打到，打到了计分，没打到不计分（普通怪普攻，可躲）
- **瞬时检测**：不数秒，选手出拳的瞬间裁判看有没有打到（宠物攻击，无预警）

**生活比喻 2（本游戏应用）**：红尾蝎翘尾巴 0.5 秒（前摇预警）后刺过来：
- 改前（必中）：你跑开了也挨刺 8 血——你会觉得"我都跑开了还挨打不公平"
- 改后（前摇后检查）：你跑出 40 像素就 miss，红尾蝎刺空——你能躲掉，战斗有策略性

**反例 vs 正例**：
```
❌ 错误（硬编码 if name == "红尾蝎"）：
if monster_id == "red_scorpion":
    check_distance()  # 违反 §1.2 数据驱动
elif monster_id == "boss_001":
    guaranteed_hit()

✅ 正确（数据驱动 attack_pattern 字段）：
match _attack_pattern:
    "guaranteed_hit": _apply_damage_to_target()
    "telegraph_then_check": _check_then_attack()
# 加新怪物只改 JSON 的 attack_pattern 字段，不改代码
```

## 功能

- 让怪物攻击支持 3 种判定模式（数据驱动，不改代码切换）
- `guaranteed_hit`：必中（Boss 大招）
- `telegraph_then_check`：前摇后检查（红尾蝎，玩家跑开就 miss）
- `instant_check`：瞬时检测（宠物，攻击动画期间用 AttackArea 持续检测）
- miss 时显示灰色"miss"飘字（步骤 1 飘字系统）

---

## 运作方式

### 数据驱动字段（monsters.json / pets.json）

```json
{
  "red_scorpion": {
    "attack_pattern": "telegraph_then_check",
    "attack_range": 40,
    "telegraph_duration": 0.5
  }
}
```

### monster.gd 的 _do_attack 分支（核心代码）

```gdscript
func _do_attack() -> void:
    if _target == null or not is_instance_valid(_target):
        return
    # 温和模式检查...
    
    # 步骤 2：attack_pattern 分支
    match _attack_pattern:
        "guaranteed_hit":
            # 必中模式（Boss 用，保留原行为）
            _apply_damage_to_target()
        "telegraph_then_check":
            # 前摇后范围检查
            var dist: float = global_position.distance_to(_target.global_position)
            if dist <= attack_range:
                _apply_damage_to_target()
            else:
                # miss！显示灰色飘字
                DamageNumberManager.show_damage_number(_target.global_position, 0, "miss")
        _:
            _apply_damage_to_target()


func _apply_damage_to_target() -> void:
    # 抽出的"实际命中并扣血"逻辑
    if _target.has_method("take_damage"):
        _target.take_damage(damage)
        DamageNumberManager.show_damage_number(_target.global_position, damage, "damage")
```

### 代码执行流程（红尾蝎 telegraph_then_check 模式）

```
红尾蝎走到 attack_range=40 内 → 启动预警（_is_telegraphing=true, _telegraph_timer=0.5）
→ 0.5 秒后预警结束 → 调用 _do_attack()
→ match _attack_pattern = "telegraph_then_check"
→ 检查 dist = distance_to(_target.global_position)
   分支 A：玩家没跑（dist=30 ≤ 40）
     → _apply_damage_to_target()
     → player.take_damage(8)
     → 弹红色"8"飘字
   分支 B：玩家跑了（dist=60 > 40）
     → DamageNumberManager.show_damage_number(pos, 0, "miss")
     → 弹灰色"miss"飘字
     → 不扣血
```

### 三种模式对照表

| 模式 | 触发时机 | 判定方式 | 适用 |
|------|---------|---------|------|
| `guaranteed_hit` | 前摇结束 | 无条件命中 | Boss 大招、不可躲技能 |
| `telegraph_then_check` | 前摇结束 | 检查目标在 attack_range 内 | 红尾蝎普攻（玩家跑开就 miss）|
| `instant_check` | 攻击动画期间 | AttackArea 持续检测 | 宠物攻击（无预警）|

---

## 原理

### 为什么用 match 语句而不是 if-elif？

- `match` 是 GDScript 的多分支语法，类似 switch
- 比-if elif 链更清晰，特别是分支多的时候
- `_` 是默认分支（类似 switch 的 default）
- 编译器会在分支遗漏时给警告（如果用 enum）

### 为什么抽出 _apply_damage_to_target 函数？

- `_do_attack` 各分支都要"实际命中扣血"，重复代码
- 抽出函数后，分支只负责"判定打没打中"，函数只负责"打了扣多少"
- 单一职责原则：判定逻辑和命中逻辑分离

### telegraph_then_check 为什么用 attack_range 而不是 DamageArea？

- 步骤 2 先实现简单版（用 attack_range 距离判定）
- 步骤 3 才升级为 DamageArea 精确判定（cone 扇形 + 朝向夹角）
- 渐进式改进：先有"能 miss"，再升级为"精确判定 miss"

### 为什么宠物用 instant_check 而不是 telegraph_then_check？

- 宠物没有"前摇预警"机制（玩家不可见宠物何时出手）
- 宠物攻击是"攻击动画期间持续检测命中"（_check_pet_hit 在 _attack_timer > 0 期间调用）
- 字段统一后，未来宠物技能可改为 telegraph_then_check 或 guaranteed_hit 不改代码

---

## 优势

| 优势 | 说明 |
|------|------|
| 数据驱动 | 加新怪物只改 JSON 的 attack_pattern 字段，不改代码 |
| 可躲避 | telegraph_then_check 让玩家有"跑开躲攻击"的策略空间 |
| 模式可扩展 | 未来加"蓄力技能"等新模式只加 match 分支 |
| 与飘字系统联动 | miss 时显示灰色"miss"，玩家立刻知道"没打中" |
| 与 DamageArea 兼容 | 步骤 3 升级后用 DamageArea 精确判定，attack_pattern 仍驱动分支 |

## 使用场景

### 场景 1：普通怪普攻（可躲）

```json
// monsters.json
"red_scorpion": {
    "attack_pattern": "telegraph_then_check",
    "attack_range": 40
}
```

### 场景 2：Boss 大招（必中）

```json
// monsters.json（未来 boss_001）
"boss_001": {
    "attack_pattern": "guaranteed_hit"
}
```

### 场景 3：宠物攻击（无预警）

```json
// pets.json
"pet_001_ham_dog": {
    "attack_pattern": "instant_check"
}
```

### 场景 4：陷阱触发（未来扩展）

```gdscript
# 未来加 "trap_trigger" 模式：玩家踩到才触发
match _attack_pattern:
    "trap_trigger":
        if _player_in_trap_area():
            _apply_damage_to_target()
```

### 与其他系统对接

- **伤害飘字系统**：miss 时调用 `DamageNumberManager.show_damage_number(pos, 0, "miss")`
- **DamageArea 抽象类**：步骤 3 升级 telegraph_then_check 分支用 DamageArea 精确判定
- **数据驱动架构**：attack_pattern 字段在 monsters.json/pets.json，代码 match 分发
- **仇恨机制**：take_damage(amount, attacker) 的 attacker 参数让 attack_pattern 分支后还能触发仇恨
