# 30 - 碰撞层（Collision Layer）与碰撞遮罩（Mask）

## 本游戏实例（v2.7 准则，v2.8 回填）

- **遇到的问题**：Phase 4.4.2 宠物场景 `scenes/pet/pet.tscn` 需要分配碰撞层。docs/12-技术栈与架构.md 已经预留了"Layer 16 = 宠物（将来用）"，但 4.4.2 是首次落地使用。要决定：宠物 collision_layer 设几？collision_mask 设几？玩家撞宠物应该被挡住还是穿过去？
- **专业名词/知识点**：碰撞层（Collision Layer）、碰撞遮罩（Collision Mask）、位掩码（Bitmask）、`collision_layer=16`、`collision_mask=1`、Layer 5 = 宠物专属层
- **技术栈/代码/美术**：`scenes/pet/pet.tscn` 第 10-11 行 `collision_layer = 16` / `collision_mask = 1`、`docs/12-技术栈与架构.md` 第 91-114 行碰撞层规范段
- **应用过程**：
  1. 查 `docs/12-技术栈与架构.md` 第 91-114 行碰撞层规范，确认"Layer 5（值=16）= 宠物（将来用）"
  2. 4.4.2 把"将来用"变成"现在用"——在 `pet.tscn` 设 `collision_layer = 16`
  3. 设 `collision_mask = 1`——宠物只碰墙（Layer 1），不碰玩家（Layer 2）、不碰怪物（Layer 4）、不碰其他宠物（Layer 16）
  4. 这样玩家撞宠物穿过去（不阻挡），宠物之间也互不碰撞（避免推挤卡死）
  5. 攻击检测不靠物理碰撞，靠 Area2D（玩家 hitbox mask=4 检测怪物，将来宠物 hitbox mask=4 检测怪物）
  6. Layer 16 是地基（一次定下，未来 NPC 宠物/联机宠物都用此层），mask=1 是策略（MVP 互不碰撞，未来可升级为 Area2D 软推力）

---

## 概念

**碰撞层（Collision Layer）** = Godot 物理引擎用来控制"谁和谁会发生物理碰撞"的分层系统。每个物理节点声明"我在哪层"（Layer）和"我检测哪层"（Mask），只有双方互相匹配才会真正碰撞。

**碰撞遮罩（Collision Mask）** = "我关心哪些层"的过滤器。它决定一个物理体会不会对另一层的东西产生碰撞反应。

生活比喻：想象一栋 32 层的办公楼：
- **Layer（你在哪层）**：你在第 2 层办公
- **Mask（你跟谁有业务往来）**：你只跟第 1 层（前台）打交道

只有"A 的 Mask 包含 B 的 Layer"**且**"B 的 Mask 包含 A 的 Layer"时，A 和 B 才会真正物理碰撞（碰面）。否则就算在同一栋楼也互不干扰（各走各的）。

**本游戏专属例子 1（生动形象）**：把碰撞层想象成《太阳之下》世界的"隐身楼层"。玩家住在 2 楼，怪物住在 4 楼，宠物住在 16 楼。每个楼层的住户只能看见"前台"（1 楼的墙）——所以大家走到墙边都被挡住，但互相看不见对方（穿过去）。只有"监控摄像头"（Area2D hitbox）能跨楼层看见特定楼层的人——玩家 hitbox 摄像头只看 4 楼（怪物），所以攻击能命中怪物但不会打到自己（2 楼）或宠物（16 楼）。

**本游戏专属例子 2（本游戏应用）**：Phase 4.4.2 宠物 `pet_001_ham_dog` 落地 Layer 16。在 `pet.tscn` 第 10-11 行写：
```ini
collision_layer = 16    ; 宠物在 Layer 5（值=16）
collision_mask = 1      ; 宠物只检测墙（Layer 1，值=1）
```
效果：
- 玩家（layer=2, mask=1）走过来 → 玩家 mask=1 不含宠物 layer=16 → 玩家穿过去
- 宠物（layer=16, mask=1）走过来 → 宠物 mask=1 不含玩家 layer=2 → 宠物穿过去
- 宠物撞墙 → 宠物 mask=1 含墙 layer=1 → 被墙挡住
- 怪物 hitbox（mask=4）→ 不含宠物 layer=16 → 不会误伤宠物（除非玩家手动关温和模式）

**反例 vs 正例**：
```
❌ 错误（不分层）：所有人都在第 1 层，所有人都互相碰撞
   → 玩家被怪物挤到墙角，所有方向卡死
   → 攻击 hitbox 检测到自己（玩家也在第 1 层）

✅ 正确（分层）：墙=1层 / 玩家=2层 / 怪物=4层 / 宠物=16层
   → 玩家只碰墙（mask=1），怪物只碰墙（mask=1），宠物只碰墙（mask=1）
   → 玩家/怪物/宠物互不碰撞，永不卡墙角
   → hitbox 设 mask=4，只检测怪物，不会打自己也不会误伤宠物
```

## 功能

- 控制物理体之间是否发生碰撞阻挡
- 控制Area2D 检测器关心哪些物理体（攻击命中、拾取、触发器）
- 让不同类型的物体各走各的路，不互相干扰
- Godot 提供 32 个层（Layer 1~32），用位掩码表示

## 运作方式

### 在 Godot 4 里怎么设置

**方法 1：编辑器里设**——选中节点 → 右侧检查器 → PhysicsBody2D → Collision → Layer 勾选对应的层 / Mask 勾选要碰撞的层。

**方法 2：在 .tscn 文件里设**（本项目用法）：
```text
[node name="Player" type="CharacterBody2D" parent="."]
collision_layer = 2    ; 玩家在第 2 层
collision_mask = 1     ; 只碰第 1 层（墙）
```

**方法 3：在代码里设**：
```gdscript
collision_layer = 2  # 玩家在第 2 层
collision_mask = 1   # 只碰第 1 层
_hitbox_area.collision_mask = 4  # hitbox 只检测第 4 层（怪物）
```

### 代码逐行执行流程（本项目实际代码）

**场景：玩家按 J 攻击，hitbox 检测怪物**

```gdscript
# === 第 1 步：_ready() 里创建 hitbox 并设置 mask ===
_hitbox_area = Area2D.new()                    # 新建一个看不见的检测区
_hitbox_area.name = "AttackHitbox"
add_child(_hitbox_area)                         # 挂到玩家节点下

_hitbox_area.collision_mask = 4                 # 关键：mask=4，只检测 layer 4（怪物）
                                                # 玩家自己在 layer 2，hitbox 不会检测到玩家

_hitbox_area.monitoring = false                 # 平时关闭，只在攻击瞬间开

# === 第 2 步：_physics_process() 里按 J 触发攻击 ===
if Input.is_action_just_pressed("attack") and _attack_cd_timer <= 0:
    _start_attack()                             # 进入攻击状态
    # _start_attack 里：_attack_timer = 0.2（攻击持续0.2秒）

# === 第 3 步：攻击持续期间，每帧检测命中 ===
if _attack_timer > 0:
    _hitbox_area.monitoring = true              # 开启检测
    _update_hitbox_position()                   # hitbox 跟着面向方向移动
    if not _attack_hit:                         # 还没命中过
        _check_hit()                            # 检测这一帧有没有打到怪

# === 第 4 步：_check_hit() 里用 mask 过滤 ===
func _check_hit() -> void:
    # get_overlapping_bodies() 返回当前和 hitbox 重叠的所有物理体
    # 但因为 mask=4，只会返回 layer 4 的物体（怪物），不会返回玩家自己
    var bodies: Array = _hitbox_area.get_overlapping_bodies()
    for body in bodies:
        if body == self:                        # 三重保险：代码层面再排除自己
            continue
        if body.has_method("take_damage"):      # 确认对方能受伤
            body.take_damage(attack_damage)      # 调用怪物的受伤方法
            _attack_hit = true                   # 标记本次攻击已命中
            break                                # 只打一个
```

**为什么 `collision_mask = 4` 能过滤？** Godot 每物理帧（60fps）遍历所有 Area2D，对每个 Area2D 检查它的 mask 和场景中每个物理体的 layer 是否有交集。`mask=4`（二进制 100）只和 `layer=4`（二进制 100）匹配，所以 `get_overlapping_bodies()` 只返回怪物，不返回玩家（layer=2，二进制 010）。

### 位掩码原理

Godot 的 layer 和 mask 是**位掩码（bitmask）**，不是简单的层号。每一层对应一个二进制位：

| 层编号 | 十进制值 | 二进制 |
|-------|---------|--------|
| 1 | 1 | 00000001 |
| 2 | 2 | 00000010 |
| 3 | 4 | 00000100 |
| 4 | 8 | 00001000 |
| 5 | 16 | 00010000 |

如果一个物体同时在 layer 1 和 3：`collision_layer = 1 + 4 = 5`（二进制 00000101）。

判断"A 和 B 是否碰撞"= 用按位与（AND）运算：
```
A.mask & B.layer != 0  且  B.mask & A.layer != 0
```
例：玩家 mask=1（00000001），怪物 layer=4（00000100），`1 & 4 = 0`，不碰撞。

## 原理

Godot 物理引擎每帧（60fps）做两件事：
1. **物理碰撞检测**：对每对物理体检查 layer/mask 是否互相匹配。匹配的才计算碰撞响应（阻挡、滑动）。
2. **Area2D 重叠检测**：对每个开启 `monitoring` 的 Area2D，检查它的 mask 和场景中物理体的 layer 是否匹配。匹配的进入 `get_overlapping_bodies()` 列表。

**关键区别**：
- `CharacterBody2D` / `StaticBody2D` 的 layer/mask 控制**物理碰撞**（会不会互相挡住）
- `Area2D` 的 layer/mask 控制**检测范围**（关心哪些层的东西进入区域），Area2D 本身不阻挡任何东西

这就是为什么玩家和怪物可以"互不物理碰撞"（CharacterBody2D 的 mask 互不包含），但 hitbox（Area2D）仍然能"检测到怪物"（Area2D 的 mask 设为怪物的 layer）。

## 优势

| 优势 | 说明 |
|------|------|
| 一行配置解决卡墙 | 不分层时怪物挤玩家到墙角会全方向卡死，分层后互不碰撞永不卡 |
| hitbox 不打自己 | mask=4 只检测怪物层，玩家在 layer 2 自然被过滤 |
| 性能优化 | 物理引擎跳过不匹配的层对，减少碰撞计算量 |
| 可扩展 | 加新系统只需分配新 layer 编号，不动现有代码 |
| 策略可升级 | 地基（层编号规范）不变，策略（互不碰撞→软推力）可随阶段升级 |

## 使用场景

### 本项目碰撞层规范（架构地基）

```
Layer 1（值=1）：墙体、地面、障碍物
Layer 2（值=2）：玩家
Layer 4（值=4）：怪物、NPC
Layer 8（值=8）：道具、可交互物体（将来用）
Layer 16（值=16）：宠物（将来用）
```

所有物理体的 mask 规范：
- 墙：mask=0（不需要碰任何东西）
- 玩家：mask=1（只碰墙）
- 怪物：mask=1（只碰墙）
- 玩家 hitbox Area2D：mask=4（只检测怪物）

### 墙角卡死的三种解决方案 + 架构地基三问论证

#### 方案 A：碰撞层分离（当前采用）

**原理**：玩家 layer=2 mask=1，怪物 layer=4 mask=1，互不物理碰撞。攻击靠 Area2D/距离检测。

| 架构地基三问 | 回答 |
|-------------|------|
| ① MVP 够用吗？ | ✅ 一行配置，零额外代码，零 bug 风险 |
| ② 加 NPC/宠物/联机时重构吗？ | ✅ NPC 加 layer 4，宠物加 layer 16，规范不变 |
| ③ 升级时地基改吗？ | ✅ 策略可升级为方案 C，碰撞层编号规范不动 |

#### 方案 B：手动墙滑行修正

**原理**：保留物理碰撞，`move_and_slide()` 后遍历碰撞法线，只阻挡墙方向。

| 架构地基三问 | 回答 |
|-------------|------|
| ① MVP 够用吗？ | ✅ 但代码复杂，角点碰撞易出 bug |
| ② 加系统重构吗？ | ⚠️ 每种新物理体都要写墙滑行逻辑，维护成本高 |
| ③ 升级地基改吗？ | ❌ 从 B 升级到 C 要重写物理逻辑，策略不可平滑升级 |

#### 方案 C：Area2D 软推力（未来升级目标）

**原理**：怪物身上加 Area2D，检测到玩家重叠时施加推力速度（不靠物理碰撞，靠代码推）。

| 架构地基三问 | 回答 |
|-------------|------|
| ① MVP 够用吗？ | ✅ 但需额外写 Area2D 推力代码 + 调参（推力大小/方向/衰减） |
| ② 加系统重构吗？ | ✅ 可抽象为 `SoftForceArea` 组件，被击退/推挤/宠物避让复用 |
| ③ 升级地基改吗？ | ✅ 本身是升级终点，地基（碰撞层规范）和 A 相同 |

#### 地基结论：为什么 MVP 选 A 而非 C

**方案 A 和 C 的地基完全相同**——都是"碰撞层编号规范"。C 不是"地基更完善"，而是"策略更接近最终形态"。

但从"减少未来重构"角度，C 确实能省一次升级工作。然而 MVP 原则是"先跑通再优化"：
- A = 零代码零风险，MVP 首选
- C = 多代码多调参，联机版需要推挤感时再升级
- **升级路径已规划**：A → C 时，碰撞层规范不动，只在怪物脚本加 Area2D + 推力逻辑

**直接做 C 的风险**：MVP 阶段只有玩家-怪物一种交互，软推力组件的抽象需要多个用例才能设计好。过早抽象 = 违反"先跑通再优化"原则。所以 MVP 用 A，但在 docs/12 预留 C 的接口规划。

## 常见错误

### 错误 1：hitbox 检测到自己
- **原因**：Area2D 没设 mask，默认检测所有层，包括玩家自己的层
- **修复**：设 `_hitbox_area.collision_mask = 4`（只检测怪物层）+ 代码 `if body == self: continue`
- **三重防护**：mask 过滤 + 代码排除 + 碰撞层分离

### 错误 2：怪物把玩家挤到墙角卡住
- **原因**：玩家和怪物在同一层，互相物理碰撞
- **修复**：玩家 layer=2 mask=1，怪物 layer=4 mask=1，互不碰撞

### 错误 3：怪物穿墙
- **原因**：怪物的 mask 不包含墙的 layer
- **修复**：确保怪物 mask 包含墙的 layer（mask=1）

## 历史教训

> 2026-07-18 Phase 4.1：玩家和怪物都在 layer 1（未分层），导致三个 bug 同时出现：①hitbox 检测到自己（按 J 自己打自己飘伤害数字）②怪物物理碰撞挡住无法走进攻击距离（怪物不攻击）③被挤到墙角全方向卡死。根本原因是知识库 27 旧规范说"MVP 阶段简化没分层"。修复：建立碰撞层规范（1=墙/2=玩家/4=怪物）+ hitbox mask=4 + 代码排除自己。同时修正知识库 27 统一到新规范。
