# 33 - 状态机骨架（State Machine Skeleton）

## 本游戏实例（v2.7 准则）

- **遇到的问题**：Phase 4.4.2 宠物系统启动。宠物需要在不同行为间切换——待机不动、跟随玩家、攻击怪物、温和模式无敌。如果不组织代码结构，所有逻辑会堆在 `_physics_process` 里，越加越乱。需要一种"清晰划分行为边界"的写法。
- **专业名词/知识点**：状态机（State Machine）、状态（State）、状态切换（State Transition）、`match` 语句、状态函数骨架
- **技术栈/代码/美术**：GDScript 的 `match` 语句、`_state: String` 字符串变量、`_state_idle()` / `_state_follow()` / `_state_attack()` 三个函数骨架（pet.gd 第 54-79 行）
- **应用过程**：
  1. 在 pet.gd 顶部声明 `var _state: String = "idle"`，默认待机状态
  2. 在 `_physics_process` 里用 `match _state:` 根据当前状态调用对应函数
  3. 三个状态函数 `_state_idle/_state_follow/_state_attack` 在 4.4.2 仅 `pass`（空骨架），4.4.4-4.4.6 填充逻辑
  4. 通过 `set_state("follow")` 等方法切换状态
  5. 这样未来加新状态（如"逃跑"）只需加一个 `_state_flee` 函数 + match 加一行，不动现有代码

---

## 概念

**状态机** = 一种把"对象在不同情况下的行为"分门别类装进不同格子的设计模式。对象同一时间只在某一个格子里（一个状态），满足条件就跳到另一个格子（状态切换）。

**生活比喻 1（生动形象）**：交通信号灯有三个状态——红、黄、绿。同一时间只能亮一个颜色，时间到了自动切到下一个。司机看到红灯就停、绿灯就行——信号灯不需要知道"现在应该亮什么颜色"的逻辑，它只要按状态显示对应颜色。

**生活比喻 2（本游戏应用）**：宠物 `pet_001_ham_dog`（火腿狗）有三种行为状态：
- **idle（待机）**：玩家没动，宠物原地站着
- **follow（跟随）**：玩家走远了，宠物追上去保持 2-3 身位
- **attack（攻击）**：附近有怪物，宠物冲过去咬

宠物同一时间只做一件事，不会一边跟一边咬。状态切换由"玩家距离"和"怪物距离"触发。

**反例 vs 正例**：
```
❌ 错误（堆在一起）：
func _physics_process(delta):
    if 玩家走远了:
        # 跟随逻辑 50 行
    elif 附近有怪物:
        # 攻击逻辑 80 行
    elif 玩家没动:
        # 待机逻辑 20 行
    # 加新行为 → 继续堆 elif，越来越乱

✅ 正确（状态机骨架）：
func _physics_process(delta):
    match _state:
        "idle":    _state_idle(delta)    # 50 行封装在函数里
        "follow":  _state_follow(delta)  # 80 行封装在函数里
        "attack":  _state_attack(delta)  # 20 行封装在函数里
        # 加新行为 → 加一个状态函数 + match 一行
```

---

## 功能

- 把复杂行为按"状态"分门别类，每个状态一个函数
- 同一时间只执行一个状态的逻辑，互不干扰
- 状态切换通过修改 `_state` 变量触发，下一帧自动跳到对应函数
- 加新行为 = 加新状态函数 + match 加一行，不改现有逻辑

---

## 运作方式

### Phase 4.4.2 骨架代码（pet.gd 第 21-79 行）

```gdscript
# === 状态变量 ===
var _state: String = "idle"  # 当前状态，默认待机

# === 主循环：根据状态分发 ===
func _physics_process(delta: float) -> void:
    if _is_dying:
        return

    if Input.is_action_just_pressed("pet_toggle_mode"):
        toggle_gentle_mode()

    match _state:                    # 关键：match 语句按 _state 值分发
        "idle":
            _state_idle(delta)       # 调用待机函数
        "follow":
            _state_follow(delta)     # 调用跟随函数
        "attack":
            _state_attack(delta)     # 调用攻击函数

# === 三个状态函数骨架（4.4.2 仅 pass，4.4.4-4.4.6 填充）===
func _state_idle(delta: float) -> void:
    pass                              # 4.4.2 留空，4.4.4 实现"原地待机+检测玩家走远"

func _state_follow(delta: float) -> void:
    pass                              # 4.4.2 留空，4.4.4 实现"move_and_slide 跟随+保持 2-3 身位"

func _state_attack(delta: float) -> void:
    pass                              # 4.4.2 留空，4.4.6 实现"检测怪物+攻击+冷却"

# === 状态切换方法 ===
func set_state(new_state: String) -> void:
    _state = new_state                # 改 _state 变量，下一帧 match 自动跳到新函数
```

### 代码逐行执行流程

**场景：玩家走远，宠物从 idle 切到 follow**

```
帧 1：
  _state = "idle"
  match _state → 命中 "idle" → 调用 _state_idle(delta)
  _state_idle 内部：检测到玩家距离 > 80 像素 → 调用 set_state("follow")
  _state 变成 "follow"

帧 2（下一帧）：
  _state = "follow"
  match _state → 命中 "follow" → 调用 _state_follow(delta)
  _state_follow 内部：move_and_slide 跟随玩家
```

**关键点**：状态切换不会立刻执行新状态逻辑，要等下一帧。因为 `_state_idle` 已经在执行中，改 `_state` 后函数返回，下一帧 match 才会读到新值。

---

## 原理

GDScript 的 `match` 语句类似其他语言的 `switch-case`，按变量值跳到对应分支。`match _state` 让代码每帧根据 `_state` 字符串调用不同函数。

**为什么用字符串而不是枚举？**
- MVP 阶段字符串简单直观，调试时 `print(_state)` 直接看状态名
- Godot 4 的 GDScript 枚举也支持，但零基础用户读字符串更容易
- 后续如果状态多了（>5 个），可以升级为枚举，地基（状态机结构）不动

**为什么状态函数命名带 `_state_` 前缀？**
- 函数名一眼能看出是状态处理函数
- 避免和普通方法混淆（如 `toggle_gentle_mode` 不是状态函数）
- Godot 自动补全输入 `_state_` 会列出所有状态函数

**状态机的本质**：把"无序的 if-elif 链"升级为"有序的状态分发"。每个状态是独立的函数，内部只关心自己那部分逻辑，不互相干扰。

---

## 优势

| 优势 | 说明 |
|------|------|
| 行为边界清晰 | 每个状态一个函数，逻辑不互相挤压 |
| 加新行为不改旧代码 | 加 `_state_flee` 函数 + match 一行，现有状态不动 |
| 调试容易 | `print(_state)` 直接看当前状态，定位 bug 快 |
| 状态切换可控 | 所有切换都走 `set_state()`，加日志/校验只改一处 |
| 升级平滑 | 字符串→枚举→正式 StateMachine 类，渐进升级，骨架不动 |

## 使用场景

### 场景 1：宠物 AI（Phase 4.4.2 当前实现）

宠物有 idle/follow/attack 三个状态。`_physics_process` 用 match 分发。4.4.4 实现 follow 逻辑，4.4.6 实现 attack 逻辑。

### 场景 2：怪物 AI（monster.gd 已隐式使用）

monster.gd 的 `_ai_melee_charger` 函数内部已有状态机雏形：
- `_is_telegraphing = true/false` 是隐式状态
- `_is_dying = true/false` 是另一个隐式状态

未来如果怪物行为复杂化（>3 个状态），可以升级为显式状态机骨架（同 pet.gd）。

### 场景 3：玩家攻击状态（player.gd 已隐式使用）

player.gd 的 `_attack_timer > 0` 是隐式"攻击中"状态，`_is_dying` 是隐式"死亡中"状态。未来玩家行为复杂化时也可升级。

### 与其他系统对接

- **DataManager**：状态机不直接读 JSON，但状态函数内部可能调用 `DataManager.get_monster(id)` 获取目标数据
- **InputManager**：状态切换可能由输入触发（如玩家按 Z 切换温和模式，间接影响宠物是否进入 attack 状态）
- **Signal**：状态切换可发信号 `state_changed(old, new)` 通知 UI 更新（如温和模式图标）
