# 49 - 物品体系与数据驱动

> 上接 [[03-数据驱动架构与JSON工作原理]]，落地到 [[06-单例模式与Autoload]] 的 DataManager。

## 本游戏实例（v2.7 强制段）

**遇到的问题**：
Phase 4.5 启动"物品体系 + Inventory"模块时发现 3 个地基空洞：
1. `data/items.json` 的 `usable / material / food / sub_prof / skill_book` 5 个分类都是空字典 `{}`，无实际物品
2. `data/items.json` 缺 `weapon / armor` 两个分类（schema_example 标了 8 类，实际只建了 6 类）
3. `data/equipment.json` 的 `_data` 是空字典，无装备属性
4. `DataManager.get_item()` 只查 6 类，漏 weapon/armor
5. 红尾蝎掉落的 `material_leather_001` 在 items.json 中无定义，邮件附件/NPC 商店/烹饪系统都依赖物品 ID 必须有数据

**专业名词/知识点**：
- **物品体系 8 类**：important / usable / weapon / armor / food / material / sub_prof / skill_book（见 `docs/06-物品体系.md`）
- **数据驱动**：物品属性写在 JSON，代码只读不写
- **跨文件字段引用**：weapon/armor 类物品用 `equip_id` 字段关联到 `equipment.json` 的同 ID 装备属性
- **Dictionary.merged()**：GDScript 4 内置方法，合并两个字典，后者覆盖前者

**技术栈/代码**：
- `data/items.json`（物品通用属性：name/category/rarity/description/tradeable）
- `data/equipment.json`（装备专属属性：attack/defense/durability_max/skill_id/repair_cost_base）
- `scripts/core/data_manager.gd` 三个接口：
  - `get_item(id)` —— 查 items.json 8 类，返回通用属性
  - `get_equipment(id)` —— 查 equipment.json._data，返回装备属性
  - `get_item_full(id)` —— 合并两者，仅 weapon/armor 类合并

**应用过程**：
1. 看 `docs/06-物品体系.md` 确认设计真值（8 类 + 装备耐久规则）
2. 看 `items.json` 现状（只有 important 类有 7 个物品，其他全空）
3. 给 items.json 加 5 个 MVP 必需物品：
   - usable: `potion_hp_small`（小回血药水，30 HP / 15s CD）
   - weapon: `sword_basic`（新手铁剑，关联 equip_id）
   - armor: `armor_cloth`（布衣，关联 equip_id）
   - food: `food_grilled_fish`（烤鱼，20 HP）、`food_stew`（炖菜，35 HP）
   - material: `material_leather_001`（红尾蝎掉落）、`fish_common`、`meat_raw`、`veg_carrot`
4. 给 equipment.json `_data` 加 sword_basic 和 armor_cloth 的装备属性（attack/defense/durability_max/repair_cost_base）
5. 扩展 `DataManager.get_item` 从查 6 类改为查 8 类（加 weapon/armor）
6. 新增 `get_equipment` 和 `get_item_full` 接口

## 概念

### 1. 生动形象例子（图书馆分区）

把 `items.json` 想象成一座图书馆，按内容分成 8 个分区：

| 分区名 | 装什么 | 借阅规则 |
|--------|--------|---------|
| important 区 | 稀世古籍（太阳印记） | 不外借、不可丢 |
| usable 区 | 工具书（药水） | 可借 99 本 |
| weapon 区 | 武器图鉴 | 可借，但要登记 |
| armor 区 | 护甲图鉴 | 可借，但要登记 |
| food 区 | 食谱 | 可借 30 本 |
| material 区 | 原料标本 | 可借 99 本 |
| sub_prof 区 | 副职业工具 | 可借 |
| skill_book 区 | 技能教材 | 可借 |

每本书有唯一编号（item_id，如 `sword_basic`）。借书时图书管理员（`DataManager.get_item`）按编号顺序扫 8 个分区，找到就返回，找不到返回空字典。

### 2. 本游戏实际应用例子

玩家击败红尾蝎后，触发掉落流程：

```
怪物死亡 → 读取 monsters.json["red_scorpion"].drops
  → [{"item_id": "material_leather_001", "chance": 0.4}]
  → 40% 概率掉落 → 调 InventoryManager.add_item("material_leather_001")
    → InventoryManager 先调 DataManager.get_item("material_leather_001")
      → 查 items.json.material → 找到 → 返回:
        {name:"普通皮革", category:"material", rarity:"white",
         description:"从红尾蝎等小型怪物身上剥下的皮革...",
         tradeable:true, stack:99}
    → 加入背包 material 槽位
```

玩家装备新手铁剑：

```
打开背包 → 点 sword_basic → 调 EquipmentManager.equip("sword_basic")
  → 调 DataManager.get_item_full("sword_basic")
    → get_item 查 items.json.weapon → 拿基础信息
    → category=="weapon" → 调 get_equipment 查 equipment.json._data
    → 拿 {attack:10, defense:0, durability_max:100, repair_cost_base:10}
    → merged() 合并两份字典
  → 返回完整数据 → 装备到武器槽 → 攻击力 +10
```

## 功能

1. **统一物品数据源**：游戏里所有物品的属性（名字/描述/稀有度/可交易/堆叠上限/装备属性）全部在 JSON
2. **加新物品零改代码**：JSON 加一行 + DataManager 自动可查
3. **跨文件字段引用**：weapon/armor 用 `equip_id` 关联到 equipment.json，避免数据冗余
4. **三接口分层查询**：
   - 只想要基础信息 → `get_item(id)`
   - 只想要装备属性 → `get_equipment(id)`
   - 想要完整合并数据 → `get_item_full(id)`

## 运作方式（含具体例子）

### DataManager 启动加载流程

```
游戏启动
  ↓
DataManager._ready()
  ↓
for f in _FILES:  # 16 个 JSON 文件名
  load_file(f)    # 读 res://data/{f}.json → 解析 → 存入 _cache[f]
  ↓
_cache = {
  "items": {...8 类物品...},
  "equipment": {"_meta":..., "_data": {...2 件装备...}},
  "monsters": {...},
  ...
}
```

### 三接口查询示例

```gdscript
# 1. 查基础信息（任何物品都可查）
var item: Dictionary = DataManager.get_item("sword_basic")
# 返回: {name:"新手铁剑", category:"weapon", rarity:"white",
#        description:"新手村铁匠铺最便宜的铁剑...",
#        tradeable:true, equip_id:"sword_basic"}

# 2. 查装备属性（仅装备类有数据）
var equip: Dictionary = DataManager.get_equipment("sword_basic")
# 返回: {name:"新手铁剑", category:"weapon", rarity:"white",
#        attack:10, defense:0, durability_max:100,
#        skill_id:"", tradeable:true, repair_cost_base:10}

# 3. 查合并数据（最常用，装备类专用）
var full: Dictionary = DataManager.get_item_full("sword_basic")
# 返回: items.json + equipment.json 合并后的完整字典
# 包含 description（来自 items）+ attack/durability_max（来自 equipment）
```

### get_item_full 的合并逻辑

```gdscript
func get_item_full(id: String) -> Dictionary:
    var item: Dictionary = get_item(id)         # 先查 items.json
    if item.is_empty():                          # 物品 ID 不存在
        return {}
    if item.get("category", "") in ["weapon", "armor"]:
        var equip: Dictionary = get_equipment(id)  # 查 equipment.json
        if not equip.is_empty():
            return item.merged(equip)              # 合并（equip 覆盖 item 同名字段）
    return item                                    # 非装备类直接返回
```

**为什么用 merged**：
- items.json 的 weapon 物品有 `name`、`equip_id`、`tradeable`
- equipment.json 的 sword_basic 有 `name`、`attack`、`durability_max`、`repair_cost_base`
- merged 后两份合并，name 字段相同取后者（这里都是"新手铁剑"所以无差异），其他字段叠加

## 原理

### 1. 数据驱动 vs 硬编码

```
❌ 硬编码（违反 §1.2）：
if item_id == "sword_basic":
    attack = 10
    durability = 100
elif item_id == "armor_cloth":
    defense = 5
    durability = 100

✅ 数据驱动（合规）：
var data = DataManager.get_equipment(item_id)
attack = data.get("attack", 0)
durability = data.get("durability_max", 0)
```

加新装备时：
- 硬编码：改 .gd 文件 + 重新编译
- 数据驱动：equipment.json 加一行

### 2. 单一职责原则（SRP）

为什么不把所有装备属性都塞进 items.json？

| 方案 | 优点 | 缺点 |
|------|------|------|
| 单文件（items.json 全装） | 查询简单 | 一个文件爆炸大，数据职责混杂 |
| 双文件（items + equipment 分离） | 职责清晰 | 需要 get_item_full 合并 |

选了双文件：
- `items.json` 装通用属性（任何物品都有的 name/desc/rarity）
- `equipment.json` 装装备专属属性（attack/defense/durability/skill_id）
- 跨文件用 `equip_id` 字段做"指针"

### 3. 跨文件字段引用

```
items.json:
  "weapon": {
    "sword_basic": {
      "name": "新手铁剑",
      "equip_id": "sword_basic"  ← 指针，指向 equipment.json
    }
  }

equipment.json:
  "_data": {
    "sword_basic": {              ← 被 items.json 的 equip_id 指向
      "attack": 10,
      "durability_max": 100
    }
  }
```

`equip_id` 是显式的"指针声明"，让代码知道这个物品有装备属性需要去另一个文件查。

---

## 优势

1. **加新物品零改代码**：JSON 加一行，DataManager 自动可查
2. **平衡调整不重新编译**：改 JSON 数值，重启游戏即生效
3. **多人协作解耦**：数值策划改 JSON，程序员改 .gd，互不冲突
4. **职责清晰**：通用属性 vs 装备属性分开存放，不会一个大文件爆炸
5. **跨文件引用灵活**：weapon/armor 类才合并，其他类直接返回，无性能浪费
6. **未来扩展空间**：加新分类只需在 items.json 加段 + get_item 的 for 循环加分类名

## 使用场景（含对接）

### 当前对接（MVP 阶段）

| 系统 | 调用什么接口 | 用途 |
|------|-------------|------|
| InventoryManager | `get_item(id)` | 加入背包前验证 ID 有效 |
| EquipmentManager | `get_item_full(id)` | 装备时拿完整属性 |
| NPC 商店 | `get_item(id)` | 显示物品名字/描述/价格 |
| 怪物掉落 | `get_item(drop.item_id)` | 掉落后入包前验证 |
| 邮件附件 | `get_item(mail.item_id)` | 附件领取时验证 |
| 烹饪系统 | `get_item(material_id)` | 验证材料 ID |

### 数据流示例

```
NPC 商店挂物品 ID 列表:
  ["sword_basic", "armor_cloth", "potion_hp_small", "fish_common", "meat_raw"]

玩家点击购买 sword_basic:
  → NPC 商店代码调 DataManager.get_item("sword_basic")
  → 显示: 名字="新手铁剑", 描述="新手村铁匠铺最便宜的铁剑..."
  → 玩家确认购买 → InventoryManager.add_item("sword_basic")
  → 钱币扣除（未来经济系统）
```

### 未来扩展（联机版）

- 服务器权威下，物品 ID 由服务器下发，客户端只渲染
- `get_item_full` 改为从服务器拉数据，接口不变（§0.5 第三问：升级地基不改）
- 加暴击/闪避/抗性等新字段时，只需在 equipment.json 加字段，代码用 `.get("crit", 0)` 兼容

## 反例 vs 正例对照

| 反例（违规） | 正例（合规） |
|------|------|
| `if name == "sword_basic": attack = 10` | `attack = DataManager.get_equipment(id).attack` |
| 所有装备属性塞进 items.json 一个文件 | items.json 装通用 + equipment.json 装专属 |
| `get_item` 只查 6 类漏 weapon/armor | 查 8 类全覆盖 |
| 改物品数值要重新编译游戏 | 改 JSON 重启即生效 |
| 装备 ID 用整数 1/2/3 | 用字符串 ID "sword_basic" 可读可搜索 |
