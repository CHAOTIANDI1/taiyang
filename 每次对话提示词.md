## 强制执行清单（违反任意一条 = 任务失败，必须补做）

### 第 0 层 · 架构地基优先原则（§0.5，凌驾所有规则）

任何技术选型（碰撞层、数据结构、状态机、组件划分、物理策略）必须先答"架构地基三问"，三问全 PASS 才可选：

1. **MVP 够用吗？**（当前阶段能跑通）
2. **加 NPC/宠物/联机/新系统时要重构吗？**（地基可扩展）
3. **要升级时地基（规范/接口）要改吗？**（策略可升级，地基不动）

论证必须写入对应知识库笔记的"使用场景"段 + `docs/12-技术栈与架构.md`。
只选"最简单"不论证 = 违规。

区分两个概念：
- **架构地基**：一次定下、长期不变的规范（如碰撞层编号分配）
- **实现策略**：可随阶段升级的做法（如互不碰撞 → 软推力）

### 第 1 层 · 闸门 A 任务前清单（不打印 = 禁止开干）

每次开始执行任务前，AI 必须先打印以下清单给用户看：

```
任务：<描述>
适用规矩：
  - §X.X 内容 → 需要做什么
  - ...
触发条件核对：
  - 是否触发知识库补全 §1.4 第 9 项？
  - 是否触发三项一致 §1.6？
  - 是否触发闸门 C 差异审计？
  - 是否触发架构地基三问 §0.5？
确认完毕，开始执行。
```

### 第 2 层 · 执行中实时同步

每个子任务完成立即同步四样东西，不批量补、不延后：
- 代码（`.gd` / `.tscn`）
- 设计文档（`docs/`）
- 知识库（`知识库/`）
- 数据文件（`data/*.json`）

### 第 3 层 · 闸门 B 完成审计 7 问（7 项全 ✅ 才算完成）

每次完成任务后，AI 必须打印：

1. 改了 code 同步改 docs 了吗？（§1.6 三项一致）
2. 改了 JSON 同步改 code 了吗？
3. AGENTS.md §2.1 MVP 范围有没有越界？
4. §1.8 能力总目录有没有未登记的新系统？
5. §1.4 知识库有没有该补未补的笔记？
6. 三项一致性验证通过吗？（设计/数据/代码对应一致）
7. commit 信息按 §1.6 格式吗？（`[类型] 内容，已同步 XX.md 和 XX.json`）

### 第 4 层 · 闸门 C 差异审计（实际做法与原定不一致时必走）

5 问全答 + 用户确认后才动，事后回写真值：
1. 实际做法与原定哪里冲突？
2. 原定为什么行不通？
3. 新做法是否更优？证据是什么？
4. 改规矩 vs 改做法，代价哪个小？
5. 是否同步回写 docs/ 或 AGENTS.md？

## 知识库 7 段标准（§1.4，每篇笔记必须全合格）

每篇笔记统一格式：
1. **概念**：含生活类比，零基础读得懂
2. **功能**：讲清楚做什么
3. **运作方式**：含具体例子 + **代码逐行执行流程**（不能只写配置方法）
4. **原理**：讲为什么
5. **优势**：讲好处
6. **使用场景**：含项目对接 + **架构地基三问论证**（技术选型类必答）
7. **关联段**：双链到相关笔记（`[[文件名]]`），形成知识网

- 每个概念必须有生活类比
- 每个机制必须有"反例 vs 正例"对照
- 末尾必须有关联段

## 零容忍规则

- 滑坡 = 任务失败，必须立即补做
- 只修代码不解释 = 违规
- 只解释不补知识库 = 违规
- 用专业术语解释给小白 = 无效解释
- 选型无论证 = 违规
- 闸门 A 不打印就开干 = 违规
- 闸门 B 不做 7 问就声称完成 = 违规
- 三项不一致（docs/数据/代码）= 违规

## 报错三步流程（§1.4 第 9 项，不可跳过）

每次用户贴报错或修 bug：
1. **解释根因**：用生活比喻讲清楚为什么发生
2. **修复**：明确指出哪行哪词错了，改成正确版本
3. **补知识库**：按 7 段标准新建或补充笔记，并更新索引
