### 已确认渲染规格

| 项 | 值 | 说明 |
|----|----|------|
| 渲染帧率 | **60fps** | 画面刷新率， Godot 默认锁定 |
| 动画帧率 | **8~12fps** | 帧切换速度，独立于渲染 |
| 目标分辨率 | **1280×720 (16:9)** | Steam 标准最低分辨率 |
| 像素密度 | **32×32** | 角色/怪物统一规格 |
| 缩放策略 | 整数倍缩放（×2/×3） | 保持像素清晰，避免半像素模糊 |
| 全屏模式 | **支持全屏自适应** | Godot 内置 stretch mode + windowed-fullscreen toggle，保持像素密度 |

### 经验值曲线（MVP）

- Lv1→2 需 ~100 XP，Lv14→15 需 ~2000+ XP
- 递增曲线（`NEXT_LEVEL_XP = BASE * level^GROWTH_RATE`，公式具体值待 MVP 调试）
- Lv15 后未转职锁经验，必须去小城镇转职才能继续升级
- MVP 玩家可达最高 Lv15
- 此值不是"严格遵守"，MVP 制作时可调，**先定"递增 + Lv15 转职"作为基本约束**

### 时间系统架构

| 项 | MVP（单机） | 联机版 |
|----|-----------|--------|
| `GameTime` 单例（Autoload） | ✅ 实现 | ✅ 复用 |
| 时间来源 | 本地系统时间换算 | 服务器统一时间 |
| 太阳历起点 | 426 年 3 月 18 日 06:26（固定常量） | 同左 |
| UI 显示 | 状态栏可展开的日期时间面板 | 同左 |
| 写死在代码里的等待时长 | ❌ | ❌ |
| `data/timers.json` 配置文件 | ✅ 留接口 | ✅ 使用 |

**时间系统统一叫"太阳历"**：单机用它，联机也用它。唯一区别是单机版从本地系统时间换算，联机版从服务器统一时间取。

### 现实等待接口

- 种植成熟、炼金转化这类依赖现实时间的系统，等待时长**全部写在 `data/timers.json`**
- 调整数值 = 改 JSON 一个数字，不动代码、不影响存档、不影响平衡
- 数值待依赖此接口的系统真正实现时再调

## 架构地基优先原则（§0.5，凌驾所有规则）

任何技术选型必须先答"架构地基三问"，三问全 PASS 才可选：

1. **MVP 够用吗？**（当前阶段能跑通）
2. **加 NPC/宠物/联机/新系统时要重构吗？**（地基可扩展）
3. **要升级时地基（规范/接口）要改吗？**（策略可升级，地基不动）

**区分两个概念**：
- **架构地基**：一次定下、长期不变的规范
- **实现策略**：可随阶段升级的做法

## 碰撞层规范（架构地基）

本项目碰撞层编号分配（一次定下，长期不变）：

| Layer | 十进制值 | 用途 |
|-------|---------|------|
| 1 | 1 | 墙体、地面、障碍物 |
| 2 | 2 | 玩家 |
| 3 | 4 | 怪物、NPC |
| 4 | 8 | 道具、可交互物体（将来用） |
| 5 | 16 | 宠物（将来用） |

Mask 规范：
- 墙：mask=0（不碰任何东西）
- 玩家：mask=1（只碰墙）
- 怪物：mask=1（只碰墙）
- 玩家 hitbox Area2D：mask=4（只检测怪物）

**v2.9 夯实地基：AttackArea mask 数据驱动**：
- 玩家 hitbox mask：从 `characters.json` 的 `attack_target_mask` 字段读（默认 4=怪物层）
- 宠物 AttackArea mask：从 `pets.json` 的 `attack_target_mask` 字段读（默认 4=怪物层）
- 怪物 AttackArea mask：从 `monsters.json` 的 `attack_target_mask` 字段读（默认 18=玩家(2)|宠物(16)）
- 联机版加 PvP 时改 JSON 即可，不改代码

**实现策略（可升级）**：
- MVP 阶段：方案 A（碰撞层分离，互不物理碰撞，攻击靠 Area2D/距离检测）
- 联机版升级目标：方案 C（Area2D 软推力，怪物身上加 Area2D 检测重叠时施加推力）
- 升级路径：A → C 时，碰撞层规范（地基）不动，只在怪物脚本加 Area2D + 推力逻辑

详见知识库 `知识库/30-碰撞层与碰撞遮罩.md` 和 `知识库/47-AttackAreaFactory与mask数据驱动.md`。

## 三条地基规矩（从第一行代码起遵守）

### 规矩 1：游戏逻辑与画面显示分离

```
❌ 错误：角色攻击 → 直接扣怪物血条 → 播放受伤动画
✅ 正确：角色攻击 → 计算伤害值 → 通知"怪物"扣血 + 通知"画面"播放动画
```

游戏规则（伤害计算、等级检查、物品能否使用）放独立脚本，**永远不直接操作画面**。将来搬上服务器不用改代码。

### 规矩 2：数据表驱动，不写死代码

```
❌ 错误：if monster_name == "红尾蝎": damage = 15
✅ 正确：读取 monsters.json，找到"红尾蝎"那行，取 damage 字段
```

JSON 文件就是数值表。加新怪物 = JSON 加一行，不改代码。

### 规矩 3：存档用统一接口

不管本地存 JSON 还是未来服务器存 PostgreSQL，调用 `SaveManager.save()` 的方式不变。底层实现可换，上层代码不动。

## 联机版启动时考虑清单（v2.9 评估登记）

经过 §0.5 三问 + §1.7 4 问 + §15 新技术引用审计 5 问评估，以下 4 项在 MVP 阶段**不必要**，但联机版启动时**强烈建议做**：

| 项 | 类型 | MVP 不做的理由 | 联机版必做的理由 |
|----|------|--------------|----------------|
| EventBus 事件总线 | 新技术 | 现有 signal 机制够用，违反 §19 过早抽象 | 客户端→服务器→所有客户端广播事件的标准做法 |
| IDamageable 接口抽象 | 新技术 | GDScript 用 `has_method` duck typing 就够，改继承关系太重 | 服务器权威下所有可受伤单位必须严格接口约束 |
| 数据验证框架 | 新技术 | MVP 阶段 push_warning 手动验证就够，违反 §19 | 服务器权威时数据校验必须严格，防止客户端篡改 |
| combat_formulas.json | 新 JSON | MVP 没有暴击/闪避，公式直接扣血 | 联机版加暴击/闪避/抗性时，公式数据驱动便于平衡调整 |

**注意**：本清单登记在 docs/12 而非 AGENTS.md §1.4 数据文件清单——因为 MVP 阶段不实现，不算"地基"，联机版启动时再走 §1.7 新增系统协议登记到 AGENTS.md。

## 知识库 AI-RAG 5 层升级方案（v3.1 落地）

### ⚠️ v3.1 知识库独立声明（凌驾性原则）

> **知识库是用户人生脑子，独立于《太阳之下》游戏项目**：
> - 知识库记录用户人生所有知识（游戏设计 + 人生哲理 + 跨项目经验 + 读书笔记等），是用户一辈子的资产
> - 知识库技术栈（Python）独立于游戏技术栈（GDScript），完全分离
> - 游戏代码（`scripts/`）不引用知识库工具；知识库工具（`tools/`）不引用游戏代码
> - 做下一个项目（Web/App/AI/任何领域）时，知识库工具能原样搬走，不用重写
>
> 详见 AGENTS.md §1.4 顶部声明 + §20.11 知识库技术栈独立性原则。

### 设计目标

把"死的档案柜"升级为"活的思考伙伴"，但**不替代闸门**——AI 提建议、闸门做决策、用户做终审。详见 AGENTS.md §20。

### 5 层架构

| 层 | 名称 | 本期落地 | 解决的问题 |
|---|------|:------:|----------|
| 第 1 层 | 语义索引 | ❌ 延后联机版 | 关键词搜不到（语义搜索） |
| 第 2 层 | AI 问答 | ✅ | 知识库不会"开口说话" |
| 第 3 层 | 自动巡检 | ✅ | 笔记质量塌方（断链/重复/过时/不达标/三项不一致） |
| 第 4 层 | 跨笔记推理 | ❌ 延后联机版 | 笔记间隐含关联看不见 |
| 第 5 层 | 项目同步 | ✅ | 代码改了笔记没改（三项不一致自动化） |

### 本期落地的 3 层详情

#### 第 2 层 AI 问答（协议层，无代码）

- 用户用大白话提问，AI 结合知识库 + 代码 + docs 生成答案
- **强制 4 段输出格式**：直接答案 / 出处标注 / 反例 vs 正例 / 关联推荐
- 盲区暴露：知识库没记载时主动说"建议走闸门 A 触发补全"
- 触发：用户问"这是什么/为什么/怎么做"时强制走 4 段格式

#### 第 3 层 自动巡检（Python 脚本 + 模板，v3.1 改）

- 巡检脚本：`tools/kb_inspector.py`（v3.1 改为 Python，命令行运行 `python tools/kb_inspector.py`）
- 巡检 5 项：
  1. 断链检测（`[[文件名]]` 目标不存在）→ 🔴 高
  2. 重复定义（多篇笔记解释同一概念）→ 🔴 高
  3. 过时检测（代码改了笔记 30 天没动）→ 🟡 中
  4. 准则不达标（缺"本游戏实例"/"本实例"段 / 概念段少于 2 个例子）→ 🟡 中
  5. 三项不一致（笔记描述与实际代码不符）→ 🔴 高
- 触发时机：每次会话开始 + 用户主动喊 + 里程碑完成
- 报告归档：`知识库/.ai-reports/巡检-YYYYMMDD-HHMM.md`（.gitignore 排除）

> v3.1 变更：原 v3.0 用 GDScript + @tool + EditorScript 写在 `scripts/tools/kb_inspector.gd`，v3.1 改为 Python 写在 `tools/kb_inspector.py`。理由：知识库独立于游戏项目，技术栈应独立（详见 AGENTS.md §20.11）。

#### 第 5 层 项目同步（Python 脚本 + 模板，v3.1 改）

- 同步脚本：`tools/sync_advisor.py`（v3.1 改为 Python，命令行运行 `python tools/sync_advisor.py`）
- 功能：解析 git log，找出"代码改了但相关笔记 30 天没动"的情况
- 触发时机：每次 git commit 后 + 用户主动喊"同步检查"
- 报告归档：`知识库/.ai-reports/同步-YYYYMMDD-HHMM.md`

> v3.1 变更：原 v3.0 用 GDScript + @tool + EditorScript 写在 `scripts/tools/sync_advisor.gd`，v3.1 改为 Python 写在 `tools/sync_advisor.py`。

### 知识库技术栈独立性原则（v3.1 新增）

**核心立场**：知识库工具技术栈必须独立于游戏技术栈。

| 项 | 游戏代码（scripts/） | 知识库工具（tools/） |
|----|--------------------|---------------------|
| 语言 | GDScript | Python 3.8+ |
| 打包 | 进 Steam 包 | 不进 Steam 包 |
| 引用 data/ | ✅ 引用 | ❌ 不引用 |
| 引用 assets/ | ✅ 引用 | ❌ 不引用 |
| 引用对方 | ❌ 不引用 tools/ | ❌ 不引用 scripts/（只读不引用，用于第 5 层检查代码改动） |
| 运行环境 | Godot 编辑器 / 游戏运行时 | Python 命令行 |

**5 条独立理由**：
1. 跨项目复用（未来做别的项目时整个 tools/ 目录可以原样搬走）
2. 技术栈分离（互不影响，改一个不会让另一个崩）
3. 未来加 embedding 平滑过渡（同语言 Python）
4. 符合 §0.5 架构地基三问（地基 Python 独立于游戏引擎）
5. Python 是 AI/数据科学通用语言（学了用一辈子）

### 协同机制（AI + 闸门）

```
用户问 → AI 答（4 段格式，标注出处）
     → AI 建议补笔记（不自动写）
     → 走闸门 A 打印清单
     → 走闸门 C 差异审计（如果是大改）
     → 用户确认
     → AI 写笔记（按 v2.7 准则 8 段）
     → 走闸门 B 第 8 问验收（准则符合度）
     → 入库 + 更新索引
     → 走第 3 层巡检（30 天后复查）
```

### 永远不引入

- ❌ AI 自动写入笔记（必须走闸门 C 用户确认）
- ❌ AI 自动删除笔记（§6.1 不删除 Git 历史）
- ❌ 云端知识库（隐私 + 项目独立性）
- ❌ 通用知识库 AI（必须绑定本项目知识）
- ❌ 知识库工具用 GDScript 写（v3.1 起，锁死 Godot 不允许）

### 联机版启动时补做

| 项 | 类型 | 联机版必做理由 |
|---|------|--------------|
| 第 1 层 语义索引 | 新技术 | 笔记破 200 篇，Ctrl+F 不够用。技术选型：BGE-M3（开源本地 embedding）+ SQLite-vec（轻量向量库），都是 Python 生态 |
| 第 4 层 跨笔记推理 | 新技术 | 复用第 1 层索引，挖掘笔记间隐含关联，建议双链 |

**v3.1 关键**：第 1/4 层引入时仍用 Python（BGE-M3 + SQLite-vec 都是 Python 库），与本期第 3/5 层技术栈统一，平滑过渡，不分裂技术栈。
