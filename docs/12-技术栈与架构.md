### 已确认渲染规格

| 项 | 值 | 说明 |
|----|----|------|
| 渲染帧率 | **60fps** | 画面刷新率， Godot 默认锁定 |
| 动画帧率 | **8~12fps** | 帧切换速度，独立于渲染 |
| 目标分辨率 | **1280×720 (16:9)** | Steam 标准最低分辨率 |
| 像素密度 | **32×32** | 角色/怪物统一规格 |
| 缩放策略 | 整数倍缩放（×2/×3） | 保持像素清晰，避免半像素模糊 |
| 全屏模式 | **支持全屏自适应** | Godot 内置 stretch mode + windowed-fullscreen toggle，保持像素密度 |

### 经验值曲线（MVP）

- Lv1→2 需 ~100 XP，Lv14→15 需 ~2000+ XP
- 递增曲线（`NEXT_LEVEL_XP = BASE * level^GROWTH_RATE`，公式具体值待 MVP 调试）
- Lv15 后未转职锁经验，必须去小城镇转职才能继续升级
- MVP 玩家可达最高 Lv15
- 此值不是"严格遵守"，MVP 制作时可调，**先定"递增 + Lv15 转职"作为基本约束**

### 时间系统架构

| 项 | MVP（单机） | 联机版 |
|----|-----------|--------|
| `GameTime` 单例（Autoload） | ✅ 实现 | ✅ 复用 |
| 时间来源 | 本地系统时间换算 | 服务器统一时间 |
| 太阳历起点 | 426 年 3 月 18 日 06:26（固定常量） | 同左 |
| UI 显示 | 状态栏可展开的日期时间面板 | 同左 |
| 写死在代码里的等待时长 | ❌ | ❌ |
| `data/timers.json` 配置文件 | ✅ 留接口 | ✅ 使用 |

**时间系统统一叫"太阳历"**：单机用它，联机也用它。唯一区别是单机版从本地系统时间换算，联机版从服务器统一时间取。

### 现实等待接口

- 种植成熟、炼金转化这类依赖现实时间的系统，等待时长**全部写在 `data/timers.json`**
- 调整数值 = 改 JSON 一个数字，不动代码、不影响存档、不影响平衡
- 数值待依赖此接口的系统真正实现时再调

## 为什么选 Godot

- 完全免费、无限制、无版税
- 轻量（~90MB），打开就能干
- 2D 是原生强项
- GDScript 像 Python，零基础友好
- AI 代码辅助效果好
- 内置联机节点（未来扩展）

## 为什么选 GDScript 而非 C#

- GDScript 是 Godot 母语，所有教程和社区基于它
- 学习曲线低（像 Python）
- 2D 像素 RPG 性能无瓶颈
- 有需要时个别模块可用 GDExtension 调 C++ 优化

## 三条地基规矩（从第一行代码起遵守）

### 规矩 1：游戏逻辑与画面显示分离

```
❌ 错误：角色攻击 → 直接扣怪物血条 → 播放受伤动画
✅ 正确：角色攻击 → 计算伤害值 → 通知"怪物"扣血 + 通知"画面"播放动画
```

游戏规则（伤害计算、等级检查、物品能否使用）放独立脚本，**永远不直接操作画面**。将来搬上服务器不用改代码。

### 规矩 2：数据表驱动，不写死代码

```
❌ 错误：if monster_name == "红尾蝎": damage = 15
✅ 正确：读取 monsters.json，找到"红尾蝎"那行，取 damage 字段
```

JSON 文件就是数值表。加新怪物 = JSON 加一行，不改代码。

### 规矩 3：存档用统一接口

不管本地存 JSON 还是未来服务器存 PostgreSQL，调用 `SaveManager.save()` 的方式不变。底层实现可换，上层代码不动。

## 存储演进路线

```
MVP：本地 JSON 文件 + SQLite（零配置，文件即数据库）
  ↓ 联机版时迁移
联机：PostgreSQL + Redis（语法几乎一样，迁移成本低）
```

## 未来架构概览

```
- 客户端：Godot 4 (GDScript)
- 服务器：Nakama（含账号/匹配/聊天/排行榜）
- 数据库：PostgreSQL
- 缓存：Redis
- 部署：Docker Compose

需要服务器权威的系统：战斗结算、交易市场、经济流通、宠物捕捉概率
保留客户端处理的系统：渲染、输入、UI、本地存档缓存
```
